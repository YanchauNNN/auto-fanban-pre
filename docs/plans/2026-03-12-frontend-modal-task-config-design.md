# 前端弹窗式任务配置设计

**日期：** 2026-03-12

**范围：** 把当前以前端主页长表单为中心的交付流程，重构为“主页面工作台 + 文件选择器 + 主配置弹窗 + 翻版次级弹窗”的交互方式，同时保留现有 API 的真实边界，并为 `audit_check` / `audit_replace` 预留前端结构。

---

## 目标

用以下结构替换当前需要依赖主页面长滚动的录入方式：

- 一个更紧凑的主页面工作台
- 一个小型 `上传 DWG` 按钮
- 一个主任务配置弹窗，用于文件摘要与主参数录入
- 一个翻版次级弹窗，用于 `source_project_no` 与 `target_project_no`
- 一个严格按 `required / required_when / required: false` 分区的表单结构

新的设计必须满足两点：

- 对 `deliverable` 保持与现有真实 API 兼容
- 对 `audit_check` 与 `audit_replace` 只预留 UI、状态与数据流，不伪造后端能力

---

## 已确认决策

### 主交互

- 主页面保留系统状态、最近任务、简要说明与任务入口。
- 上传 DWG 不再使用主页大面积拖拽区。
- 主入口是一个小型 `上传 DWG` 按钮。
- 点击按钮后调用原生文件选择器。
- 文件选择完成后，进入主任务配置弹窗。

### 弹窗层级

- 主配置弹窗负责：
  - 已选文件摘要
  - 项目号识别摘要
  - 主参数区
  - 高级选项区
  - 次级任务开关
- `纠错` 与 `翻版` 不再作为首页三张主入口卡。
- `纠错` 与 `翻版` 是互斥的次级任务开关。
- 勾选 `翻版` 后，再打开更小的翻版次级弹窗录入翻版专属参数。

### 字段分区

- 只有 `required: false` 字段进入高级选项区。
- 所有 `required: true` 字段必须停留在主参数区。
- 所有当前命中 `required_when` 的字段，也必须停留在主参数区。
- 允许字段在联动条件变化后，在“主参数区 / 高级选项区”之间动态移动。

### 翻版规则

- `source_project_no` 默认取本次上传 DWG 的识别结果。
- `source_project_no` 允许用户手工修改，以应对特殊情况。
- `target_project_no` 由用户输入。
- 两个字段都采用输入框。
- 输入框下方显示推荐项目号标签。
- 推荐值来源：
  - 本次上传 DWG 的文件名识别结果
  - 参数规范中的项目号枚举
- 校验规则：
  - `source_project_no` 必填
  - `target_project_no` 必填
  - `source_project_no != target_project_no`

### 识别值的使用方式

- 浏览器侧允许根据文件名前 4 位数字做项目号预识别。
- 前端识别结果仅用于预览、联动与推荐，不是最终事实来源。
- 最终仍以后端 API 的真实推断与校验为准。

### 视觉方向

- 采用蓝色工程工作台风格，不走纸张米黄色方向。
- 首页是紧凑工具台，不是大卡片表单台。
- 所有长内容滚动都发生在弹窗内部，不能再依赖主页面滚轮。
- 主弹窗宽而稳，翻版弹窗更小、更聚焦。

### 测试方式

- 需要做真实浏览器交互测试。
- 需要覆盖鼠标点击、键盘输入、文件上传、弹窗开关与弹窗内部滚动。
- 需要补一份给测试人员使用的 markdown 说明文档，放在 `documents/` 下。

## 信息架构

### 主页面

- 顶部工具带：
  - 品牌
  - `上传 DWG` 主按钮
  - 轻量系统状态
- 中部区域：
  - 最近批次摘要
  - 最近任务列表与筛选
- 辅助区域：
  - 简短操作说明

主页面应该更像任务队列工作台，而不是一张直接填表的业务单。

### 主任务配置弹窗

- 顶部：
  - 标题
  - 已选文件数量
  - 项目号识别摘要
  - 关闭按钮
- 左侧：
  - 已选文件列表
  - 重新选择文件
  - 文件级提示与警示
- 右侧：
  - 主参数区
  - 次级任务开关
  - 高级选项折叠区
- 底部：
  - 取消
  - 打开翻版设置
  - 提交任务

### 翻版次级弹窗

- `source_project_no`
- `target_project_no`
- 推荐项目号标签
- 校验提示区
- 保存 / 取消操作区

---

## 数据流

### 文件选择

1. 用户点击 `上传 DWG`
2. 原生文件选择器返回 `File[]`
3. 前端保存文件列表到弹窗状态
4. 前端按文件名推导临时项目号建议
5. 打开主配置弹窗，并显示文件摘要与识别结果

### 任务意图映射

- 未勾选次级任务：`deliverable`
- 勾选 `纠错`：`audit_check`
- 勾选 `翻版`：`audit_replace`

这里只改变前端交互组织方式，不改变后端任务类型语义。

### 翻版默认值

- 翻版弹窗从当前识别摘要中读取默认项目号
- `source_project_no` 先自动带入，再允许人工改写
- 保存后，把翻版参数回写到主弹窗状态

### 校验

- 主表单校验继续走 schema 驱动
- 高级选项默认不阻塞，除非该字段被 `required_when` 激活
- 翻版弹窗有自己独立的阻塞式校验

---

## 与当前 API 的兼容方式

- `deliverable` 继续走真实可用接口
- `audit_check` 与 `audit_replace` 只在前端层面预留状态、UI 与 payload builder
- 在后端接口未开放前，这两条分支不能伪装成真实可提交能力，必须有清晰限制提示

这样可以保证前端结构先走通，但不虚构后端行为。

---

## 需要控制的风险

- 动态字段分区不能把当前必填字段误收进高级选项
- 文件名识别不能被错误当成权威来源
- 多文件混合项目号批次必须尽早提示
- 两层弹窗在 `1366x768` 下必须仍可用
- 滚动必须锁在弹窗内部，不能回退到页面长滚动

---

## 验收标准

- 主页面不再承担长表单录入
- 用户可以通过“文件选择器 + 主配置弹窗”完成交付任务配置
- 高级选项中只包含可选字段
- 翻版弹窗支持可编辑的 `source_project_no / target_project_no` 与推荐标签
- `纠错` 与 `翻版` 互斥
- 真实 `deliverable` 提交流程不被破坏
- `documents/` 下存在给测试人员使用的前端测试说明
