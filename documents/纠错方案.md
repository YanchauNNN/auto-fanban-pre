# 翻版实装与块替换研究落地方案

## 摘要
- 当前仓库状态已经适合开始做“翻版”后端实装：前端 `frontend/` 里已有翻版配置 UI 和结果展示预埋，后端 `audit_replace` 任务族、词库规则、纠错扫描链、部署包/API 框架都已存在；缺口主要是 **`mode=replace` 的执行链** 和 **“翻版后再出图” 的任务编排**。
- 本轮范围锁定为：**先做文本翻版**，块替换只输出工程结论和后续设计，不在本轮一起实现。
- 交付模式锁定为两种：
  - `仅翻版`：输出 `replaced.dwg + report.xlsx + report.json`
  - `同步出图和翻版`：先翻版，再把翻版后的 DWG 作为 `deliverable` 的输入继续出图；最终同时保留 `replaced.dwg/report` 和 `package.zip/IED计划.xlsx`
- 本轮默认 **不做“纠错+翻版”同包组合**；纠错继续走现有 `mode=check`。

## 关键改动

### 1. 后端/API：把 `audit_replace` 从“仅 check”扩成“check + replace”
- 扩展 `POST /api/jobs/audit-replace`，支持 `mode=replace`。
- `mode=replace` 的 `params_json` 固定为：
  - `source_project_no`
  - `target_project_no`
  - `run_deliverable`
  - `deliverable_params`：仅当 `run_deliverable=true` 时出现，内容就是当前出图表单参数
- 校验规则固定：
  - `source_project_no`、`target_project_no` 必填
  - 两者不能相同
  - `run_deliverable=true` 时，`deliverable_params` 走现有 `DocParamValidator`
  - 不再对翻版任务做 `project_no` 推断/回退；翻版必须显式选源项目和目标项目
- 任务编排固定：
  - `仅翻版`：每个上传文件创建一个独立 `audit_replace` 子任务
  - `同步出图和翻版`：每个上传文件创建一个 group，包含两个 child
    - `audit_replace` child：先产出翻版后的 DWG
    - `deliverable` child：以该翻版 DWG 作为输入继续走模块1-6
- 现有 job/detail 字段保持兼容，只补充真实可用的翻版产物：
  - `artifacts.replaced_dwg_available`
  - `artifacts.replaced_dwg_download_url`
  - 报告下载 URL
- 部署包只需跟随更新后的 `API/`、`backend`、`.NET Bridge DLL`、YAML 配置和 `documents_bin/词库收集.xlsx`；不引入新的系统依赖。

### 2. 翻版执行链：复用纠错底座，改为“生成 mapping -> 应用 patch -> 输出新 DWG”
- 词库来源固定为 [词库收集.xlsx](E:/project/auto-fanban-pre/documents_bin/词库收集.xlsx)，规则沿用并固化：
  - 第 1 行参与匹配
  - 第 2 行参与匹配
  - 第 3 行起参与匹配
  - 只消费“项目号列”
  - 同一行做源项目到目标项目的映射
  - 如果某文本同时存在于源项目和目标项目的可用集合中，视为 no-op，不报错也不替换
- 执行阶段固定复用现有纠错准备链：
  - ODA 转 DXF
  - 图框检测
  - 图签抽取
  - A4 grouping
  - .NET Bridge 扫描文本实体
  - ROI/internal_code 标注
- 在此基础上新增“replace mapping + apply”两段：
  - Python 端：
    - 基于词库构建 `source -> target` 的逐行映射
    - 复用当前语义判定/误报抑制规则
    - 只把允许替换的命中项生成 patch plan
  - .NET 端：
    - 新增只写工作副本的 `audit_replace_apply` 命令
    - 对可写实体做文本替换并 `SaveAs` 新 DWG
- 支持的文本实体覆盖范围与纠错保持一致：
  - `DBText`
  - `MText`
  - `AttributeReference`
  - `AttributeDefinition`
  - `MLeader`
  - 显式 `DimensionText`
  - `Table` 单元格
  - 块内与嵌套块
- 行为边界固定：
  - 不覆盖原始上传文件
  - 首版不处理 `XREF`
  - 动态字段只报告不改写表达式
  - `run_deliverable=true` 时，deliverable child 直接消费翻版后 DWG，不再回读原始上传件

### 3. 前端接入边界：我不直接改前端，本轮给前端工程师明确指令
- 当前 `frontend/` 已有 `audit_replace` 预埋，但 submit 被本地 guard 拦住；后端完成后，前端只需要接通，不需要重做交互结构。
- 给前端工程师的指令固定为：
  1. 在 `DeliverableWorkspace` 中，当 `intent === "audit_replace"` 时，不再走“接口未开放”分支。
  2. 新增 `adapter.createAuditReplace(...)`，调用 `/api/jobs/audit-replace`，传 `mode=replace`。
  3. `仅翻版` 时，提交 `source_project_no + target_project_no + files[] + run_deliverable=false`。
  4. `同步出图和翻版` 时，提交 `source_project_no + target_project_no + files[] + run_deliverable=true + deliverable_params=draft.values`。
  5. 本轮前端不允许把“纠错”和“翻版”一起勾成一个包；若用户切到翻版，则隐藏/禁用 `runAuditCheck`。
  6. 详情页对 `task_kind === "audit_replace"` 显示：
     - `replaced.dwg` 下载按钮
     - `report.xlsx/report.json` 下载按钮
     - 如果该 job/group 同时有 `deliverable` child，则继续显示 `package.zip / IED计划.xlsx`
- 这轮不要求前端改页面布局，只接 API 和按钮行为。

### 4. 块替换研究结论：可靠方案存在，但建议作为后续独立阶段
- 研究结论：
  - 最可靠的工程路线不是“爆炸块后重建”，而是 **在 side database 中定位 `BlockReference`，导入目标块定义，再创建新的 `BlockReference` 并复制变换**。
  - Autodesk 官方 `BlockReference.BlockTransform` 明确可保留 insertion / scale / rotation；`Database.Insert` / `ReadDwgFile` 可把外部 DWG 的块定义导入当前图；属性可通过 `AttributeReference` 读取和迁移。
  - 工程插件也普遍遵循这条路线：保留 layer/scale/rotation/transform，并在属性同名时迁移属性值，而不是依赖爆炸后的纯几何重建。
- 对你的需求，推荐的 v2 设计是：
  - **块识别优先级**：
    1. 块实例属性值（`AttributeReference.TextString`）
    2. 块定义内的常量 `ATTDEF/TEXT/MTEXT`
    3. 最后才用块名/EffectiveName
  - **块替换执行**：
    - 把目标块定义导入当前 DWG
    - 用原块的 `BlockTransform` 创建新引用
    - 按同名属性 tag 迁移属性值
  - **原点差异处理**：
    - 长期最稳的方式是统一“替换池”块的基准坐标
    - 若短期无法统一，再为每组替换规则定义 `basepoint_offset`，由后端在替换时补偿
- 因此建议：本轮把块替换保持在“研究完成、接口不开放”的状态，文本翻版落地后再单独做。

## 测试计划
- `mode=replace` API：
  - `source_project_no/target_project_no` 缺失返回 `422`
  - `source == target` 返回 `422`
  - `run_deliverable=false` 返回单任务批次
  - `run_deliverable=true` 返回 group + two children
- 词库映射：
  - 第 1/2/3+ 行都参与映射
  - 重复词在源/目标同时存在时不替换
  - 目标项目同行为空时记 `target_term_missing`
- 替换执行：
  - `DBText/MText/ATTRIB/ATTDEF/MLeader/Dimension/Table` 各有至少一条回归
  - 原始上传 DWG 不被覆盖
  - 产出新 DWG 可再次打开
- 组合任务：
  - `同步出图和翻版` 时，deliverable child 的输入来自 replaced DWG
  - 最终同时存在 `replaced.dwg` 与 `package.zip/IED`
- 前端联调验收：
  - 翻版模式可真实提交
  - 仅翻版时不显示出图完成文案
  - 同步出图和翻版时，列表/详情能同时看到翻版产物和交付产物
- 块替换本轮只保留研究验证，不做生产验收；验收放到下一阶段单独定义。

## 假设与默认值
- 本轮只实装**文本翻版**，块替换不进实现范围。
- `翻版` 与 `纠错` 在本轮互斥；不支持一个提交里同时开启两者。
- 前端已有“翻版”交互结构可复用，因此本轮后端接口以当前前端草图为准去对齐，不重做任务入口。
- 交付模式默认采用“同时给出翻版 DWG + 出图包”。
- 部署与终端实装不新增外部依赖，只更新已有后端/API/Bridge/配置与文档资源。
