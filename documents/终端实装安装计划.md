# 终端实装安装计划

## 目标

把 `fanban-terminal-deploy.rar` 解压到部署机后，按固定顺序完成以下工作：

1. 检查 IIS 代理前提。
2. 安装 IIS 代理组件。
3. 安装运行时依赖。
4. 初始化后端运行环境。
5. 配置 IIS 前端站点。
6. 注册后端常驻。
7. 运行健康检查。
8. 用固定样本做烟测。

本方案适用于以下场景：

- 只有部署机安装 AutoCAD 和 Office。
- 部署机不能联网。
- 其他客户端通过局域网浏览器访问部署机。

## 部署包类型

现在构建机会同时产出两类部署包：

1. 全量包：`build\fanban-terminal-deploy`
2. 局部包：`build\fanban-terminal-deploy-delta`

使用规则是：

- 第一次部署，或者你不确定离线部署机当前基线时，必须使用全量包。
- 只有在离线部署机当前内容已经匹配“上一版全量包”时，才可以只拷贝局部包做迭代更新。
- 局部包根目录下只放需要覆盖的文件，`_delta\删除清单.txt` 里会列出需要手工删除的旧文件。
- 如果局部包 `_delta\使用说明.txt` 明确提示“未检测到上一版 full 包基线”，这次更新不要只拷局部包。

## 前提

部署机需要满足：

- 已安装 AutoCAD。
- 已安装 Office。
- 已安装 IIS。
- 可以从 U 盘拷贝 `fanban-terminal-deploy.rar`。

本计划假设你把压缩包解压到：

```text
D:\FanBanServer
```

如果你的实际路径不是这个，下面命令里的 `D:\FanBanServer` 要整体替换成真实路径。

## 步骤一：解压部署包

第一次部署时，把全量压缩包：

```text
E:\project\auto-fanban-pre\build\fanban-terminal-deploy.rar
```

拷到部署机后，解压到：

```text
D:\FanBanServer
```

解压完成后，至少确认下面这些目录存在：

```text
D:\FanBanServer\frontend-dist
D:\FanBanServer\backend-runtime
D:\FanBanServer\install
D:\FanBanServer\scripts
D:\FanBanServer\documents\Resources
```

如果你是在已经部署过的离线机上做版本迭代，并且确认当前机器就是上一版全量包状态，也可以只拷贝：

```text
E:\project\auto-fanban-pre\build\fanban-terminal-deploy-delta
```

这时做法是：

1. 先把局部包里的文件按相同相对路径覆盖到 `D:\FanBanServer`。
2. 再按照 `build\fanban-terminal-deploy-delta\_delta\删除清单.txt` 删除旧文件。
3. 覆盖完成后，再从“步骤四：安装运行时依赖”继续执行校验和健康检查。

## 步骤二：先检查 IIS 代理前提

先不要急着继续部署，先确认 IIS 反代能力是否完整。

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\check_iis_proxy_prereqs.ps1
```

你会看到 JSON 输出，重点看这三个字段：

- `iis.installed`
- `url_rewrite.installed`
- `arr.installed`

### IIS 代理检查通过标准

如果结果是：

- `iis = true`
- `url_rewrite = true`
- `arr = true`

说明 IIS 反代前提已经满足，可以继续下一步。

如果 `url_rewrite` 或 `arr` 为 `false`，继续执行下一步离线安装。

## 步骤三：安装 IIS 代理组件

如果上一步显示 `URL Rewrite` 或 `ARR` 缺失，执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\install_iis_proxy_prereqs.ps1
```

这个脚本会先检查本机是否已安装：

- URL Rewrite。
- ARR。

只有缺失时才安装，不会无脑重复安装。

安装完后，再执行一次：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\check_iis_proxy_prereqs.ps1
```

确认三个字段都为 `true`。

## 步骤四：安装运行时依赖

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\install_runtime_prereqs.ps1
```

这个脚本现在会先检查本机状态：

- `.NET Framework 4.8` 或更高版本是否已安装。
- `VC++ 2015-2022 x64` 是否已安装。
- 部署包里的 `Python 3.13 embeddable runtime` 是否已展开到 `D:\FanBanServer\python-runtime`。
- 部署包里的 `NSSM` 是否已展开到 `D:\FanBanServer\install\nssm\nssm.exe`。

处理方式是：

- `.NET` 和 `VC++`：本机缺失时才调用离线安装器。
- Python：不依赖目标机系统 Python，不复制 `.venv`，而是直接把部署包里的官方 embeddable Python 解压到 `D:\FanBanServer\python-runtime`，再把随包的 `site-packages` 同步进去。
- `NSSM`：属于部署包必备组件，不再视为可选项。脚本会从随包分发的压缩包中解出 `nssm.exe`，后续服务注册直接使用部署目录里的这一份。

### 运行时安装通过标准

你可能看到以下两类结果之一：

1. 本机已安装，脚本提示跳过。
2. 本机未安装，脚本开始安装或解压。

同时应看到类似输出：

- `解压离线 Python 运行时: ...python-3.13.12-embed-amd64.zip`
- `已同步部署包 Python 依赖到: D:\FanBanServer\python-runtime\Lib\site-packages`
- `解压 NSSM 离线包: ...nssm-2.24-101-g897c7ad.zip`
- `NSSM 已就绪: D:\FanBanServer\install\nssm\nssm.exe`

只要不是“安装器不存在”“离线 Python 运行时准备失败”或“NSSM 准备失败”的提示，就说明这一步逻辑正常。

## 步骤五：准备运行环境

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\prepare_terminal.ps1
```

这个脚本会完成：

1. 再次校验并补齐运行时依赖。
2. 初始化 `storage`。
3. 探测 AutoCAD、Office、ODA、Resources。
4. 生成运行环境文件：

```text
D:\FanBanServer\scripts\runtime.env.ps1
```

这个环境文件正常情况下不需要你手工再执行一遍。

原因是：

- 后端正式启动入口 `D:\FanBanServer\scripts\start_backend.ps1` 会自动加载它。
- `register_backend_service.ps1` 最终调用的也是这个启动脚本。

只有在你想“留在当前 PowerShell 会话里，手工复用这些环境变量”时，才需要自己执行：

```powershell
. D:\FanBanServer\scripts\runtime.env.ps1
```

### 运行环境准备通过标准

这里看起来和“步骤四”有一点重叠，但这是刻意保留的：

- `install_runtime_prereqs.ps1` 负责安装/解压/同步运行时。
- `prepare_terminal.ps1` 会再次调用它，保证整机准备流程是幂等的，并且在真正生成 `runtime.env.ps1` 前再做一轮补齐后校验。

执行成功后，应至少看到：

- `runtime.env.ps1` 已生成。
- `logs\probe_target_env.json` 已生成。
- `D:\FanBanServer\python-runtime\python.exe` 存在。

如果失败，先把错误信息保存下来，再继续排查。

## 步骤六：执行深度环境检查

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\deep_check_terminal.ps1
```

这个脚本会在 `deep` 模式下重新执行一轮 Office 深检，重点验证：

1. Word 模板 PDF 导出。
2. Excel 模板 PDF 导出。
3. Word 模板复制打开。
4. Excel 模板复制打开。

输出文件在：

```text
D:\FanBanServer\logs\probe_target_env.deep.json
```

### 深度环境检查通过标准

至少应满足：

- `probe_target_env.deep.json` 已生成。
- `blocking issues = 0`。
- `Office status: pass (mode: deep)`。

如果这里失败，优先看 `probe_target_env.deep.json` 里的 `details` 字段；现在脚本会在 Excel 深检失败时保留临时目录和诊断文件，不会再只给一条模糊报错。

## 步骤七：配置 IIS 前端站点

推荐先用最简单的方式，直接通过部署机 IP 访问。

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\configure_iis_site.ps1 -SiteName FanBanTerminal -Port 80
```

这个脚本会：

1. 创建或更新 IIS 站点。
2. 让 IIS 指向：

```text
D:\FanBanServer\frontend-dist
```

此外，如果 `URL Rewrite + ARR` 都已安装，则自动启用同源 `/api` 反代。

如果你准备直接按部署机 IP 访问，不要显式传 `-HostName ""`。
在 PowerShell 里最稳的用法就是直接省略 `HostName` 参数，让脚本使用默认空值。

如果这里提示 `IIS 绑定冲突` 或 `0x800700B7`，通常说明 `80` 端口已经被另一个 IIS 站点占用，最常见就是 `Default Web Site`。
这种情况下有三个选择：

1. 先停止或修改冲突站点的绑定。
2. 把当前站点改到别的端口，比如 `8080`。
3. 继续使用非空 `HostName` 绑定。

### 访问地址怎么定

如果你这样配置：

```powershell
-Port 80
```

最终访问地址就是：

```text
http://部署机IP/
```

以后如果你想改成主机名，可以重新运行脚本，例如：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\configure_iis_site.ps1 -SiteName FanBanTerminal -Port 80 -HostName "fanban-server"
```

但这里要特别注意：

- `HostName` 只负责 IIS 主机头绑定。
- 它不会自动创建 DNS 记录，也不会自动修改 `hosts`。
- 所以“脚本执行成功”不等于“`http://fanban-server/` 已经可以直接打开”。

如果你要实际通过：

```text
http://fanban-server/
```

访问页面，至少需要满足下面之一：

1. 局域网 DNS 已经把 `fanban-server` 解析到部署机 IP。
2. 部署机和客户端都手工配置了 `hosts`。

如果没有这层解析，建议优先直接按部署机 IP 访问。

## 步骤八：注册后端常驻

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\register_backend_service.ps1
```

脚本会：

1. 默认使用部署包内的 `nssm.exe` 注册 Windows 服务并立即启动。
2. 如果这里提示找不到 `nssm.exe`，说明步骤四没有执行完成，应该先回去补齐运行时依赖。

这样后端能在部署机上常驻运行。

## 步骤九：运行健康检查

执行：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\check_health.ps1
```

重点看三部分：

1. 本地环境探针。
2. IIS 代理前提。
3. `/api/system/health`。

如果最后能看到后端健康状态，说明前后端和后端服务都已经基本通了。

## 步骤十：部署机本机打开页面

在部署机浏览器里打开：

```text
http://127.0.0.1/
```

或者：

```text
http://部署机IP/
```

确认前端页面可以正常打开。

## 步骤十一：另一台客户端验证访问

在同一局域网另一台电脑的浏览器里访问：

```text
http://部署机IP/
```

如果页面能打开，说明局域网访问已经通了。

## 步骤十二：做真实样本烟测

先用这两份固定样本验证系统：

1. `18185NE-JGS11.dwg`
2. `20261RS-JGS65.dwg`

目标是确认：

- 交付任务正常。
- 纠错任务正常。
- 联合任务正常。
- 结果可下载。

## 步骤十三：出问题时优先看哪里

如果部署过程中出问题，按这个顺序看：

1. `D:\FanBanServer\logs\probe_target_env.json`
2. `D:\FanBanServer\logs\probe_target_env.deep.json`
3. `D:\FanBanServer\scripts\runtime.env.ps1`
4. `check_iis_proxy_prereqs.ps1` 的输出。
5. `check_health.ps1` 的输出。
6. 后端服务日志和任务日志。

## 完整命令顺序

完整命令顺序如下：

```powershell
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\check_iis_proxy_prereqs.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\install_iis_proxy_prereqs.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\install_runtime_prereqs.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\prepare_terminal.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\deep_check_terminal.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\configure_iis_site.ps1 -SiteName FanBanTerminal -Port 80
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\install\register_backend_service.ps1
powershell -ExecutionPolicy Bypass -File D:\FanBanServer\scripts\check_health.ps1
```
