# 终端实装安装计划

## 1. 适用场景

本计划适用于以下真实部署环境：

- 局域网内约 100 台电脑互联
- 不能访问互联网
- 只能通过 U 盘拷贝方式部署程序
- 选定其中 1 台电脑作为部署机/执行机
- 其余电脑通过浏览器访问系统
- 目标部署机已安装 AutoCAD 和 Office
- 目标部署机未安装 Python 和 ODA
- 部署过程中不能依赖任何在线下载

本计划的目标不是“给 100 台电脑分别安装一套前后端”，而是：

- 在 1 台部署机上部署前端、后端、Worker、CAD 资源池和运行依赖
- 让其余 99 台电脑通过内网浏览器访问这台部署机提供的系统
- 让部署机即使完全离线，也能仅靠 U 盘中的部署包完成安装和启动

## 2. 明确部署结论

### 2.1 前端和后端放在哪里

前端和后端都放在同一台部署机上。

具体来说：

- 前端静态页面放在部署机的 IIS 站点目录中
- 后端 FastAPI、Worker、CAD 槽位、任务存储目录也都放在部署机上
- AutoCAD、Office、ODA、受管打印资源都只部署在这台机器上

其余客户端电脑：

- 不安装后端
- 不安装 AutoCAD
- 不安装 ODA
- 不安装 Office
- 只需要能通过浏览器访问部署机上的站点

### 2.2 推荐部署拓扑

```text
100 台客户端电脑（浏览器）
        |
        | 局域网 HTTP/HTTPS
        v
部署机（1 台）
  ├─ IIS：托管前端静态页面，反向代理 /api
  ├─ FastAPI：提供任务接口、状态接口、下载接口
  ├─ Worker：执行交付、纠错、文档生成
  ├─ AutoCAD / accoreconsole
  ├─ Office COM
  ├─ ODA File Converter
  ├─ CAD 槽位目录
  └─ storage/jobs + storage/groups + runtime
```

## 3. 部署机职责和客户端职责

### 3.1 部署机职责

部署机承担所有重任务和服务能力：

- 提供前端页面
- 提供后端 API
- 接收上传文件
- 执行 DWG -> DXF 转换
- 执行图框检测、图签提取、A4 成组
- 执行 PDF/DWG 导出
- 执行纠错扫描
- 执行文档导出
- 保存任务目录、日志目录、共享前处理目录

### 3.2 客户端职责

客户端只负责：

- 打开浏览器
- 访问部署机地址
- 上传 DWG
- 查看任务状态
- 下载结果

客户端不承担任何 CAD、ODA、Office 计算。

## 4. 离线部署包目标结构

在开发机上，需要先准备一个可以通过 U 盘拷贝到部署机的离线部署包。建议结构如下：

```text
fanban-terminal-deploy/
  frontend-dist/
  backend-runtime/
  bin/
    ODAFileConverter 25.12.0/
  documents/
    Resources/
  install/
    dotnet/
    vc_redist/
  storage/
  scripts/
  README_部署说明.md
```

### 4.1 frontend-dist

前端构建产物，供 IIS 直接托管。

### 4.2 backend-runtime

后端运行目录，至少包含：

- `API/`
- `backend/`
- `.venv/` 或可恢复的离线 Python 运行环境
- `backend/src/cad/dotnet/Module5CadBridge/bin/Release/net48/Module5CadBridge.dll`
- 运行脚本
- 服务注册脚本

说明：

- 由于目标机没有 Python，部署包必须直接携带可运行的 Python 环境
- 推荐直接携带已经构建好的 `backend/.venv`
- 不应要求目标机在线安装依赖
- `pywin32` 等 Python 库应已经包含在 `.venv` 中

### 4.3 bin/ODAFileConverter 25.12.0

ODA 必须随离线部署包一起分发，因为目标机没有 ODA。

后端运行时应从部署包内的受管路径读取 ODA，例如：

```text
D:\FanBanServer\bin\ODAFileConverter 25.12.0\ODAFileConverter.exe
```

同时应把 ODA 同目录下的运行库一并保留，包括但不限于：

- `Qt6*.dll`
- `vcruntime140.dll`
- `msvcp140.dll`
- `concrt140.dll`
- `*.tx`
- ODA 自带的 `TD_*`、`W3Dtk.dll`、`WhipTk.dll` 等文件

### 4.4 documents/Resources

这里是打印资源和校准资源的唯一源头，至少包含：

- `打印PDF2.pc3`
- `tszdef-02fc5f1cb3db4a5b8afc9cce5dca6cd1.pmp`
- `fanban_monochrome.ctb`
- `fanban_monochrome-same width.ctb`
- `打白图.ctb`
- `anchor_calibration.json`

### 4.5 install/dotnet

这里用于放离线 `.NET` 运行时安装包。建议至少携带：

- `.NET Framework 4.8` 离线安装包

说明：

- `Module5CadBridge.dll` 目标框架是 `net48`
- 目标机通常若已安装较新 AutoCAD，大概率已具备对应运行环境
- 但为了离线部署可控，仍建议随包提供 `.NET Framework 4.8` 离线安装器，避免因系统缺失运行时而中断

### 4.6 install/vc_redist

这里用于放离线 VC++ 运行时安装包。建议携带：

- `VC++ 2015-2022 x64 Redistributable` 离线安装包

说明：

- ODA 自带部分运行库，但目标机整体环境仍可能缺少 VC 运行时
- 离线安装器可作为部署时的兜底项

### 4.7 storage

可预创建空目录，也可在首次启动时自动创建：

- `storage/jobs`
- `storage/groups`
- `storage/runtime`

## 5. 部署机上的真实安装目录建议

建议在部署机上固定落到一个统一根目录，例如：

```text
D:\FanBanServer
```

目录建议如下：

```text
D:\FanBanServer
  ├─ frontend-dist
  ├─ backend-runtime
  ├─ bin
  │   └─ ODAFileConverter 25.12.0
  ├─ documents
  │   └─ Resources
  ├─ install
  │   ├─ dotnet
  │   └─ vc_redist
  ├─ storage
  ├─ logs
  └─ scripts
```

后续所有脚本、服务、IIS 站点、健康检查都围绕这个根目录执行。

## 6. AutoCAD 和 Office 的探测要求

### 6.1 AutoCAD 探测要求

AutoCAD 不随部署包分发，由部署机本机安装提供。

后端启动时必须自动探测本机可用的 AutoCAD，并优先选择：

- 版本 `>= 2010`
- 版本号最高
- `accoreconsole.exe` 真实存在

探测结果至少要落出：

- `cad_version`
- `accoreconsole_exe`
- `install_dir`

### 6.2 Office 探测要求

Office 也不随部署包分发，由部署机本机安装提供。

后端启动时应对 Office 做启动能力探测，建议至少包含：

- `Word.Application`
- `Excel.Application`
- COM 创建是否成功
- 基础导出烟测是否成功

如果后端后续补入安装路径识别，建议额外落出：

- `word_exe_path`
- `excel_exe_path`

当前部署验收标准不要求前端展示 Office 路径，但要求后端健康检查能够明确报告：

- Office 可用
- Office 不可用
- 哪个组件不可用

## 7. 离线依赖边界说明

### 7.1 Python 依赖如何处理

目标机没有 Python，因此不能要求执行：

- 在线创建虚拟环境
- 在线安装 `pip` 依赖
- 在线拉取 wheel

本计划要求：

- 在开发机上预先构建完整可运行的 `backend/.venv`
- 将该 `.venv` 直接随部署包分发
- 在部署机上直接使用这套 Python 环境启动后端

当前后端依赖至少包括：

- `fastapi`
- `uvicorn`
- `pydantic`
- `pyyaml`
- `ezdxf`
- `matplotlib`
- `openpyxl`
- `python-docx`
- `pypdf`
- `python-multipart`
- `pywin32`

### 7.2 C#/.NET 依赖如何处理

当前 CAD 桥接 DLL 为：

- `backend/src/cad/dotnet/Module5CadBridge/bin/Release/net48/Module5CadBridge.dll`

它的运行依赖边界是：

1. 必须随部署包带走的：
   - `Module5CadBridge.dll`
   - 如需调试，可一并带 `Module5CadBridge.pdb`
   - `.NET Framework 4.8` 离线安装器

2. 由部署机 AutoCAD 提供的：
   - `acdbmgd.dll`
   - `acmgd.dll`
   - `accoremgd.dll`

说明：

- 上述三个 AutoCAD 托管 DLL 不需要单独随包分发
- 它们由目标机已安装的 AutoCAD 提供
- 但 `Module5CadBridge.dll` 和 `.NET Framework 4.8` 运行环境必须纳入离线部署考虑

## 8. 具体部署步骤

### 步骤 1：在开发机准备离线部署包

在开发机执行以下工作：

1. 构建前端产物
2. 整理后端运行目录
3. 确认 `backend/.venv` 可离线运行
4. 将 ODA 拷入 `bin/ODAFileConverter 25.12.0`
5. 将 `documents/Resources` 整体纳入部署包
6. 将 `Module5CadBridge.dll` 纳入部署包
7. 准备 `.NET Framework 4.8` 和 VC++ 离线安装器
8. 准备安装脚本和启动脚本
9. 将整个 `fanban-terminal-deploy` 拷入 U 盘

### 步骤 2：在部署机安装系统级软件

在部署机上手工安装并确认：

1. AutoCAD
2. Office
3. IIS
4. 如系统缺失，安装 `.NET Framework 4.8`
5. 如系统缺失，安装 VC++ 运行时

说明：

- AutoCAD 和 Office 由部署机本机提供，不放入部署包
- ODA、Python 环境、`Module5CadBridge.dll`、打印资源、离线运行时安装器都来自离线部署包

### 步骤 3：通过 U 盘拷贝部署包到部署机

将 U 盘中的离线部署包整体复制到部署机，例如：

```powershell
D:\FanBanServer
```

不要拆散复制，保持目录结构完整。

### 步骤 4：配置 IIS 托管前端

IIS 站点应指向：

```text
D:\FanBanServer\frontend-dist
```

并配置：

- 默认文档为前端入口文件
- `/api` 反向代理到本机 FastAPI

建议目标地址：

```text
http://127.0.0.1:8000
```

### 步骤 5：启动后端

使用部署包中的 Python 环境启动后端，例如：

```powershell
D:\FanBanServer\backend-runtime\backend\.venv\Scripts\python.exe -m uvicorn API.app.main:create_app --factory --host 127.0.0.1 --port 8000
```

后续正式运行时，建议注册为 Windows 服务。

### 步骤 6：初始化 storage 和 CAD 槽位

确保以下目录可写：

- `D:\FanBanServer\storage\jobs`
- `D:\FanBanServer\storage\groups`
- `D:\FanBanServer\storage\runtime`

首次启动后，应创建：

- `storage/runtime/cad-slots/slot-01`
- `storage/runtime/cad-slots/slot-02`
- `storage/runtime/cad-slots/slot-03`
- `storage/runtime/cad-slots/slot-04`

每个槽位应包含：

- `profile/`
- `support/Plotters/`
- `support/Plotters/PMP Files/`
- `support/Plotters/Plot Styles/`
- `spool/`
- `temp/`
- `logs/`

### 步骤 7：校验受管打印资源

每个槽位 support 目录中至少应存在：

- `打印PDF2.pc3`
- `tszdef-02fc5f1cb3db4a5b8afc9cce5dca6cd1.pmp`
- `fanban_monochrome.ctb`
- `fanban_monochrome-same width.ctb`
- `打白图.ctb`

### 步骤 8：启动健康检查

调用：

```http
GET /api/system/health
```

建议至少看到：

- `ready = true`
- `storage_writable = true`
- `worker_alive = true`
- `autocad_ready = true`
- `office_ready = true`

## 9. 客户端使用方式

其余客户端电脑只需要：

1. 接入同一局域网
2. 打开浏览器
3. 访问部署机地址，例如：

```text
http://192.168.1.20/
```

或：

```text
http://fanban-server/
```

客户端无需安装：

- Python
- ODA
- AutoCAD
- Office
- 后端程序

## 10. 联调验收

### 10.1 单个联合任务验收

提交一个 DWG，并设置：

- `run_audit_check = true`

预期：

1. 只生成一个顶层 `TaskGroup`
2. 共享目录存在：`storage/groups/<group_id>/shared/<source_key>/`
3. 共享目录里只出现一份共享前处理结果
4. 子任务只有两个：
   - `deliverable_main`
   - `audit_check`

### 10.2 两个联合任务并发验收

同时提交两个顶层任务，并都启用：

- `run_audit_check = true`

预期：

1. 同时最多 2 个顶层任务 active
2. 同时最多 4 个 CAD 重执行槽位被占用
3. 两个任务的 `shared` 目录互不污染
4. 槽位日志能对应到各自 child job

### 10.3 样本烟测

必须跑这两个样本：

1. `E:\project\auto-fanban-pre\test\zhangxiaomin-feedback\18185NE-JGS11.dwg`
2. `E:\project\auto-fanban-pre\test\zhangwenze-feedback\20261RS-JGS65.dwg`

预期基线：

- `18185NE-JGS11`
  - `6 PDF + 6 DWG`
  - `001 = 9 页`
- `20261RS-JGS65`
  - `11 PDF + 11 DWG`

### 10.4 打印样式覆盖验收

分别以三种样式提交任务：

- `red_wider`
- `same_width`
- `review_white`

预期：

- 任务详情能看到正确的：
  - `plot_style_key`
  - `plot_resource_mode`
  - `ctb_path`
- `ctb_path` 与所选样式一致

## 11. 当前计划中的关键说明

### 11.1 当前是否已经摆脱 PC3/PMP 源资产依赖

没有。

当前部署计划仍然依赖：

- `documents/Resources/打印PDF2.pc3`
- `documents/Resources/tszdef-02fc5f1cb3db4a5b8afc9cce5dca6cd1.pmp`

它们是部署包中的受管源资产，必须随部署包一起分发。

### 11.2 当前是否已经做到纯槽位私有 Plotters/Profile

还没有完全收口。

当前阶段是：

- 槽位私有目录优先
- 同时保留共享镜像 fallback

这是为了保证真实业务链路在部署初期稳定可跑。

后续第三阶段验证完成后，才会考虑完全切到：

- `slot_private_only`

## 12. 建议的上线顺序

1. 先完成单部署机离线部署
2. 先让部署机本机能跑通健康检查
3. 再用另一台客户端电脑做浏览器访问验证
4. 再跑两个固定样本烟测
5. 再开启两个联合任务并发验证
6. 稳定后再开放给更多局域网用户使用
