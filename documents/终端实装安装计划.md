# 终端实装安装计划

## 目标

把当前“前端静态站点 + FastAPI 后端 + CAD 资源池 worker”的执行链安装到真实 Windows 执行节点，并完成最小可用的交付 / 纠错联合任务验证。

当前后端能力基线：

- 父任务 + 子任务（`TaskGroup`）
- 共享前处理（只做一次 ODA / 图框检测 / 图签提取 / A4 成组）
- 最多 2 个顶层任务并发
- 最多 4 个 CAD 重执行槽位
- 打印资源统一从 `documents/Resources` 管理
- 打印样式支持稳定 key：
  - `red_wider`
  - `same_width`
  - `review_white`

## 推荐部署拓扑

1. IIS

   - 托管前端静态文件
   - 反向代理 `/api` 到本机 FastAPI

1. FastAPI

   - 作为 API 层
   - 负责接收任务、调度 worker、读写状态

1. Worker

   - 与 FastAPI 同机运行
   - 负责：
     - ODA
     - AutoCAD / `accoreconsole`
     - 文档生成
     - 纠错扫描

1. 存储目录

   - `storage/jobs`
   - `storage/groups`
   - `storage/runtime/cad-slots`

## 前置依赖

### 1. Windows 执行节点

- Windows 10 / 11 或同等级 Windows Server
- 本地磁盘可写
- 能稳定运行 AutoCAD / `accoreconsole`

### 2. AutoCAD

- 安装 `>= 2010` 的 AutoCAD
- 推荐安装较高版本，并确认：
  - `accoreconsole.exe` 可用
  - 本机用户能手动启动 AutoCAD

说明：

- 后端启动时会探测本机全部 `>= 2010` 的 AutoCAD
- 默认选择最高版本

### 3. ODA

- 当前仓库依赖的 ODA 文件必须完整存在
- `DWG -> DXF` 能在本机正常执行

### 4. Office

- 若需要文档导出链路，必须安装本机 Office
- 建议与开发验证环境一致

## 安装步骤

### 步骤 1：准备代码和 Python 环境

1. 拉取最新主线代码
1. 进入仓库根目录
1. 创建 / 恢复 `backend/.venv`
1. 安装 `backend/pyproject.toml` 中依赖

建议检查：

```powershell
backend\.venv\Scripts\python.exe -V
backend\.venv\Scripts\python.exe -m pytest backend\tests\unit\test_module7_api.py -q
```

### 步骤 2：准备前端静态站点

1. 在 `frontend` 目录安装依赖
1. 执行构建
1. 将构建产物部署到 IIS 站点目录

IIS 职责：

- 提供前端页面
- 把 `/api/*` 反代到本机 FastAPI

### 步骤 3：部署 FastAPI 服务

建议先本机启动验证：

```powershell
backend\.venv\Scripts\python.exe -m uvicorn API.app.main:create_app --factory --host 127.0.0.1 --port 8000
```

后续正式化时，再注册为 Windows 服务。

### 步骤 4：初始化 storage 目录

确保以下目录可写：

- `storage/jobs`
- `storage/groups`
- `storage/runtime`

首次启动后，系统应自动创建：

- `storage/runtime/cad-slots/slot-01`
- `slot-02`
- `slot-03`
- `slot-04`

### 步骤 5：检查 CAD 槽位目录

每个槽位应包含：

- `profile/`
- `support/Plotters/`
- `support/Plotters/PMP Files/`
- `support/Plotters/Plot Styles/`
- `spool/`
- `temp/`
- `logs/`

验收重点：

1. 四个槽位目录都存在
1. 槽位 support 目录可写
1. 槽位 profile 目录可写

### 步骤 6：校验受管打印资源

每个槽位 support 目录内应存在：

- `打印PDF2.pc3`
- `tszdef-02fc5f1cb3db4a5b8afc9cce5dca6cd1.pmp`
- `fanban_monochrome.ctb`
- `fanban_monochrome-same width.ctb`
- `打白图.ctb`

说明：

- 当前阶段仍保留“共享目录镜像 fallback”作为过渡兜底
- 但源资产和槽位私有目录都应完整

### 步骤 7：启动后健康检查

调用：

```http
GET /api/system/health
```

预期至少确认：

- `ready = true`
- `storage_writable = true`
- `worker_alive = true`
- `autocad_ready = true`

## 联调验收

### 验收 1：单个联合任务

提交：

- 一个 DWG
- `run_audit_check = true`

预期：

1. 只生成一个顶层 `TaskGroup`
1. 共享目录存在：
   - `storage/groups/<group_id>/shared/<source_key>/`
1. 共享目录里只出现一份：
   - `source_converted.dxf`
   - `frames.json`
   - `sheet_sets.json`
   - `titleblock_extracts.json`
   - `audit_roi_context.json`
1. 子任务有两个：
   - `deliverable_main`
   - `audit_check`

### 验收 2：两个联合任务并发

同时提交两个顶层任务，并都启用 `run_audit_check=true`。

预期：

1. 同时最多 2 个顶层任务 active
1. 同时最多 4 个 CAD 重执行槽位被占用
1. 两个任务的 shared 目录互不污染
1. 槽位日志能对应到各自 child job

### 验收 3：样本烟测

必须跑这两个样本：

1. `E:\project\auto-fanban-pre\test\zhangxiaomin-feedback\18185NE-JGS11.dwg`
1. `E:\project\auto-fanban-pre\test\zhangwenze-feedback\20261RS-JGS65.dwg`

预期基线：

- `18185NE-JGS11`
  - `6 PDF + 6 DWG`
  - `001 = 9 页`
- `20261RS-JGS65`
  - `11 PDF + 11 DWG`

### 验收 4：打印样式覆盖

分别用三种样式提交任务：

- `red_wider`
- `same_width`
- `review_white`

预期：

- 任务详情能看到正确的：
  - `plot_style_key`
  - `plot_resource_mode`
  - `ctb_path`
- `ctb_path` 应与所选样式一致

## 故障排查重点

### 1. 如果交付和纠错仍然串行

检查：

- 前端是否还在旧逻辑里：
  - 先 `/api/jobs/batch`
  - 再 `/api/jobs/audit-replace`
- 是否真的使用了：
  - `run_audit_check=true`

### 2. 如果仍看到两次 ODA 转换

检查：

- `storage/groups/<group_id>/shared/<source_key>/prep_summary.json`
- 交付子任务和纠错子任务是否都拿到了相同的：
  - `shared_run_id`
  - `shared_prep_dir`
  - `shared_source_dxf`

### 3. 如果 CAD 资源互相污染

检查：

- child job 的 `slot_id`
- `pc3_path / pmp_path / ctb_path`
- 槽位 support 目录是否完整
- 是否还有代码在写共享 `%APPDATA%` Plotters 目录

### 4. 如果文档生成异常

检查：

- Office 是否正常安装
- 文档链是否仍使用独立的 `doc_worker_pool`

## 上线建议

第一阶段建议：

1. 先启用：
   - `max_active_groups = 2`
   - `cad_slot_count = 4`
   - `doc_slot_count = 1`
1. 先开放给内部测试用户
1. 观察：
   - 队列长度
   - CAD 槽位占用
   - 子任务失败分布
   - ODA / AutoCAD / Office 稳定性
1. 稳定后再考虑提高并发
