# 交付前端任务

## 背景

后端已完成“父任务 + 子任务 + 共享前处理”的第一版能力，目标是把一次“交付 + 纠错”的联合提交从：

- 先 `createBatch`
- 再单独 `createAuditCheck`

改成：

- 一次提交 `createBatch(..., run_audit_check=true)`
- 后端创建一个顶层 `TaskGroup`
- `TaskGroup` 下自动生成两个子任务：
  - `deliverable_main`
  - `audit_check`

这样做的目的有三点：

1. 同一个上传任务只做一次前处理
   - `DWG -> DXF`
   - 图框检测
   - 图签提取
   - A4 成组
2. 交付和纠错复用同一套共享中间产物
3. 为后端并发调度和 CAD 资源池隔离打基础

## 当前前端现状

当前前端仍然在：

- [DeliverableWorkspace.tsx](E:\project\auto-fanban-pre\frontend\src\features\deliverable\DeliverableWorkspace.tsx)

内部按以下方式提交：

1. 先调用 `adapter.createBatch(...)`
2. 如果勾选了纠错，再调用 `adapter.createAuditCheck(...)`

这会导致：

1. 前端层面天然出现“先出图、再纠错”的排队表现
2. 后端虽已有 `run_audit_check=true` 入口，但前端还没有切过去
3. 任务列表仍按“单 job”视角展示，无法表达 `TaskGroup`

## 必做改动

### 1. API 适配器改造

修改：

- [types.ts](E:\project\auto-fanban-pre\frontend\src\platform\api\types.ts)
- [httpAdapter.ts](E:\project\auto-fanban-pre\frontend\src\platform\api\httpAdapter.ts)

要求：

1. `createBatch` 新增第三个参数 `runAuditCheck?: boolean`
2. 调用 `/api/jobs/batch` 时，如果 `runAuditCheck=true`，表单里附带：
   - `run_audit_check=true`
3. 保留 `createAuditCheck(...)`
   - 仅供“纯纠错工作台”继续使用
   - 不再用于交付页面的联合提交流程

建议类型调整：

```ts
createBatch: (
  params: Record<string, string>,
  files: File[],
  runAuditCheck?: boolean,
) => Promise<CreateBatchPayload>;
```

### 2. 任务模型类型扩展

前端当前 `JobSummary / JobDetail` 还是单 job 视图，需要兼容 group。

后端现在对 group 返回的关键字段包括：

- `is_group`
- `group_id`
- `source_filenames`
- `run_audit_check`
- `child_job_ids`
- `children`（仅 detail）

要求：

1. `JobSummary` / `JobDetail` 增加 group 兼容字段
2. 不要假设所有列表项都是“单 job”
3. `jobId` 现在对 group 行来说实际就是 `group_id`

建议新增字段：

```ts
isGroup?: boolean;
groupId?: string | null;
sourceFilenames?: string[];
runAuditCheck?: boolean;
childJobIds?: string[];
children?: JobSummary[];
taskRole?: string | null;
sharedRunId?: string | null;
```

### 3. DeliverableWorkspace 提交流程改造

修改：

- [DeliverableWorkspace.tsx](E:\project\auto-fanban-pre\frontend\src\features\deliverable\DeliverableWorkspace.tsx)

当前逻辑：

- 先 `createBatch`
- 再 `createAuditCheck`

目标逻辑：

1. 如果 `draft.runAuditCheck === false`
   - 继续调用 `createBatch(params, files, false)`
2. 如果 `draft.runAuditCheck === true`
   - 直接调用 `createBatch(params, files, true)`
   - 不再单独调用 `createAuditCheck`

注意：

1. 提交成功后的 notice 文案要调整
2. 不再出现“出图任务创建成功，但纠错任务创建失败”的二次提示分支
3. `onBatchCreated(...)` 接收的 payload 中，`jobs` 现在可能是 group summary

### 4. 任务列表切换为顶层 Group 视图

当前任务列表如果直接展示 `GET /api/jobs` 的返回，需要适配：

- 列表项可能是单 deliverable job
- 也可能是一个 `TaskGroup`

要求：

1. `is_group=true` 的项显示为一张顶层任务卡片
2. 卡片内显示两个子任务 chip：
   - `交付`
   - `纠错`
3. 子任务状态从 `children` 或详情接口中读取

推荐展示：

- 顶层标题：源 DWG 文件名
- 顶层状态：`status`
- 顶层阶段：`stage`
- 子任务状态条：
  - `交付`
  - `纠错`

### 5. 详情页增加 Group 级阶段展示

详情页要从“单任务详情”扩展成“顶层 group + 子任务详情”双层视图。

对于 group：

- 展示 group 级阶段：
  - `PREP_SOURCE`
  - `DELIVERABLE_BRANCH`
  - `AUDIT_BRANCH`
  - `DOCS_AND_PACKAGE`
  - `GROUP_COMPLETE`

对于 child：

- 继续展示原有单 job 阶段和产物状态

要求：

1. 如果点开的是 group，就先显示 group 概览
2. 再显示 children 列表
3. 允许对子任务单独查看详情和下载产物

### 6. 下载入口改为按 child artifacts 展示

当前下载逻辑不要假设“所有产物都挂在单个 job 上”。

规则：

1. 顶层 group 仍可保留聚合下载入口
2. 但 UI 需要能明确区分：
   - 交付产物
   - 纠错报告

推荐：

- 顶层卡片展示：
  - `下载交付包`
  - `下载IED`
  - `下载纠错报告`
- 子任务详情页也保留自己的下载入口

## 建议联调顺序

1. 先改 `types.ts` 和 `httpAdapter.ts`
2. 再改 `DeliverableWorkspace.tsx` 的提交逻辑
3. 然后改任务列表卡片
4. 最后改详情页的 group 视图

## 联调验收

### 场景 1：仅交付

提交：

- 一个 DWG
- `runAuditCheck = false`

预期：

1. 前端仍能正常创建任务
2. 列表里出现普通交付任务
3. 下载交付包正常

### 场景 2：交付 + 纠错

提交：

- 一个 DWG
- `runAuditCheck = true`

预期：

1. 只发起一次 `/api/jobs/batch`
2. 不再发起第二次 `/api/jobs/audit-replace`
3. 列表中出现一个顶层 group 卡片
4. 卡片内能看到两个子任务：
   - `交付`
   - `纠错`
5. 详情页能看到 group 级阶段和 child 级阶段

### 场景 3：兼容纯纠错工作台

提交：

- 纯纠错工作台继续调用 `/api/jobs/audit-replace`

预期：

1. 不受这次改造影响
2. 仍能创建单独纠错任务

## 注意事项

1. 前端不要自己推导 group 结构，统一以后端返回为准
2. `run_audit_check` 只在交付工作台联合提交流程里使用
3. 纯纠错工作台不要改成 group 模式
4. 前端若暂时没做 group 详情，也至少要先切到“一次提交 run_audit_check=true”
