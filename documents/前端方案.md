# 前端方案

## 1. 结论

- 当前产物不包括前端实现，仓库里也还没有 `frontend/` 目录。
- 可以把前端实现交给另一个 AI 并行推进，但它只能负责 `frontend/` 下的页面、组件、状态管理和上传交互，不能反向定义后端任务模型。
- 当前 Web 系统要同时覆盖三类任务入口：
  - `交付处理`
  - `纠错`
  - `翻版`
- 三类入口属于同一套工程工作台，但表单、任务详情和下载产物不同。
- API 契约由后端侧维护；前端 AI 只能消费本文档冻结的接口语义，不得自行发明字段或改路由含义。

---

## 2. 当前仓库事实

- 当前仓库没有 `frontend/`
- 已确定最终形态为：
  - 后端：FastAPI
  - 前端：`frontend/` 下独立构建
  - 部署：同一台 Windows 后端机提供服务，局域网多台电脑通过浏览器访问
- 当前方案不做登录权限
- 当前方案不使用 LibreOffice / WPS
- 模块6文档阶段依赖 Office COM，且采用串行执行
- 实机离线，前端不能依赖任何 CDN
- 纠错 / 翻版属于“流水线 B”，与交付处理并列，不属于模块6文档生成

---

## 3. 前端目标与非目标

### 3.1 目标

- 提供一个美观、稳定、可直接给工程人员使用的网页工作台
- 支持多用户同时访问同一台后端机
- 支持一次上传多个 `dwg`，但后端拆成多个独立任务
- 同一界面内支持三种任务：
  - 交付处理：上传 DWG，跑到模块6，返回 `package.zip` 和 `IED计划.xlsx`
  - 纠错：按选定项目号扫描词库冲突，返回检查报告
  - 翻版：按源项目号到目标项目号映射替换文字，返回新 DWG 和替换报告
- 严格执行 `documents/参数规范.yaml` 中 `source: frontend` 的字段规则
- 仅 `required: true` 和 `required_when` 阻塞提交
- `required: false` 字段只能作为可选补充，不能阻塞业务提交
- 提供任务列表、任务详情、下载、取消、重试、健康状态

### 3.2 非目标

- 不做登录/权限/组织管理
- 不做多人协作编辑
- 不做在线预览 DWG
- 不做复杂报表系统
- 不把模块1~5的内部工程参数暴露给前端
- 不把 `split_only` 暴露给前端
- 不把纠错 / 翻版做成前端本地处理

## 4. 推荐技术栈

### 4.1 推荐实现

- `React 18 + TypeScript + Vite`
- `React Router`
- `TanStack Query`
- `Zustand` 或极简 context store
- `Axios` 或原生 `fetch` 封装
- `CSS Modules` 或 `SCSS Modules`

### 4.2 不建议

- 不要上 Next.js
- 不要上重型状态框架
- 不要引入依赖 CDN 的 UI 框架主题
- 不要直接用“白底 + 紫色按钮 + 默认系统字体”的模板化页面

### 4.3 离线约束

- 所有前端依赖必须本地构建
- 所有字体、图标、插画资产必须随项目打包
- 禁止 Google Fonts、阿里 IconFont 在线拉取、任何远程设计资源

## 5. 视觉方向

### 5.1 总体风格

- 主题方向：工程蓝图 + 纸张工单
- 目标气质：
  - 像专业工程软件，不像通用 SaaS 后台
  - 有明确秩序感，但不死板
  - 视觉要稳、准、硬朗

### 5.2 配色建议

### 5.3 字体建议

前提：必须本地打包。

- 中文正文：`Noto Sans SC` 本地子集
- 英文/数字标题：`Rajdhani` 或 `Barlow Condensed`
- 代码/编号：`JetBrains Mono`

### 5.4 动效建议

- 页面进入：顶部标题和上传卡片做轻微上浮渐入
- 任务新增：列表项高亮闪现 600ms
- 运行中进度：横向条带 + 数字百分比
- 不要堆砌微交互

### 5.5 响应式

- 主目标屏幕：
  - `1366x768`
  - `1920x1080`
- 移动端不是主战场，但页面必须能正常打开
- 小屏做单列折叠，不要让表单横向溢出

## 7. 任务入口层级

这是本次补充的核心结构，另一个 AI 不得把三个任务混成一个大表单。

### 7.1 入口定义

- `交付处理`
  - 用于上传 DWG 并跑到模块6
  - 产物为 `package.zip` 和可能存在的 `IED计划.xlsx`
- `纠错`
  - 用于按项目号做词库冲突检查
  - 产物为检查报告
- `翻版`
  - 用于按源项目号到目标项目号的同排映射替换文字
  - 产物为新 DWG 和替换报告

### 7.2 首页布局

- 首页上传区不是单一表单，而是三张任务入口卡片
- 用户先选任务入口，再显示对应表单
- 三个入口视觉上要明显区分：
  - 交付处理：偏主流程、主色更稳
  - 纠错：偏审查/质检
  - 翻版：偏迁移/改版
- 但三者仍属于同一个工程工作台，不允许做成三个完全割裂的小系统

### 7.3 表单切换规则

- 选 `交付处理` 时：
  - 显示模块6前端参数表单
  - 显示多文件上传区
- 选 `纠错` 时：
  - 只显示 `project_no`
  - 只显示多文件上传区
  - 不显示模块6文档参数
  - 不显示翻版字段
- 选 `翻版` 时：
  - 只显示 `source_project_no`
  - 只显示 `target_project_no`
  - 只显示多文件上传区
  - 不显示模块6文档参数
  - 不显示纠错字段

## 9. 前端与后端接口契约

以下接口是给另一个 AI 的默认实现契约。后端如果未来字段细节有调整，优先只改前端 `api/client.ts` 与类型定义，不要改动 UI 结构。

### 9.1 系统健康

`GET /api/system/health`

用途：

- 页面启动时检查后端是否在线
- 显示系统状态 pill

建议响应：

```json
{
  "status": "ok",
  "server_time": "2026-03-08T10:20:30+08:00",
  "ready": true,
  "storage_writable": true,
  "autocad_ready": true,
  "office_ready": true
}
```

### 9.2 表单元数据

`GET /api/meta/form-schema`

用途：

- 前端启动时读取所有 `source: frontend` 字段
- 后端统一下发 `required`、`required_when`、枚举项、格式说明
- 前端不得硬编码字段规则

建议响应结构：

```json
{
  "schema_version": "frontend-form@1",
  "upload_limits": {
    "max_files": 50,
    "allowed_exts": [".dwg"],
    "max_total_mb": 2048
  },
  "deliverable": {
    "sections": [
      {
        "id": "basic",
        "title": "基础信息",
        "fields": [
          {
            "key": "project_no",
            "label": "项目号",
            "type": "select",
            "required": true,
            "source": "frontend",
            "options": ["1818", "2016", "2026"]
          }
        ]
      }
    ]
  },
  "audit_replace": {
    "project_options": ["1418", "1818", "1916", "2016", "2026"]
  }
}
```

### 9.3 批量创建交付任务

`POST /api/jobs/batch`

`multipart/form-data`

字段建议：

- `files[]`: 多个 `dwg`
- `params_json`: JSON 字符串，存放所有前端输入参数

语义：

- 一次上传多个 `dwg`
- 后端返回一个 `batch_id`
- 每个 `dwg` 生成一个独立 `job`

建议响应：

```json
{
  "batch_id": "batch-20260308-001",
  "jobs": [
    {
      "job_id": "job-001",
      "source_filename": "A01.dwg",
      "task_kind": "deliverable",
      "status": "queued"
    },
    {
      "job_id": "job-002",
      "source_filename": "A02.dwg",
      "task_kind": "deliverable",
      "status": "queued"
    }
  ]
}
```

### 9.4 词库任务入口

当前阶段只冻结语义，不要求前端 AI 推进 API 代码。

预期语义：

- `纠错` 和 `翻版` 属于同一任务族：`audit_replace`
- 前端需按以下最小契约设计客户端层：
  - `task_kind = "audit_check"` 对应纠错
  - `task_kind = "audit_replace"` 对应翻版
  - `纠错` 请求体包含：
    - `project_no`
    - `files[]`
  - `翻版` 请求体包含：
    - `source_project_no`
    - `target_project_no`
    - `files[]`
- 前端不得把词库逻辑放到浏览器本地执行

`POST /api/jobs/{job_id}/retry`

- 是创建一个新任务


- `IED计划.xlsx` 7 天后可能过期
- 前端需要正确显示“已过期，无法下载”
- `纠错/翻版` 任务不显示 `package.zip` 和 `IED计划.xlsx`

## 12. 字段联动规则

这是前端实现最容易做错的部分。

### 12.1 任务入口联动

#### `taskKind == 'deliverable'`

- 显示模块6相关参数表单
- 显示高级选项
- 提交按钮文案使用“创建交付任务”

#### `taskKind == 'audit_check'`

- 仅显示 `project_no + dwg_files[]`
- 不显示模块6相关参数
- 不显示翻版相关字段
- 提交按钮文案使用“开始纠错检查”

#### `taskKind == 'audit_replace'`

- 仅显示 `source_project_no + target_project_no + dwg_files[]`
- 不显示模块6相关参数
- 不显示纠错字段
- 提交按钮文案使用“开始翻版”

### 12.2 1818 / 非1818切换

#### `project_no == '1818'`

- 显示：
  - `subitem_name_en`
  - `album_title_en`
- 隐藏并禁用：
  - `cover_variant`
- 同时前端提示：
  - “1818 项目使用专用封面与目录模板”

#### `project_no != '1818'`

- 显示：
  - `cover_variant`
- 隐藏并禁用：
  - `subitem_name_en`
  - `album_title_en`

### 12.3 `ied_status` 联动

#### `ied_status == '发布'`

- 展开“IED 编审链”分组
- `work_hours` 变为必填
- 所有编审链字段变为必填

#### `ied_status != '发布'`

- 编审链字段可折叠
- 不阻塞提交

### 12.4 `file_category`

- 必须是下拉框
- 选项来自后端 schema，不允许自由输入

### 12.5 `ied_design_type`

- 下拉
- 选项来自模板 Sheet4

### 12.6 `ied_responsible_unit` / `ied_discipline_office`

- 下拉
- 建议：
  - 先选 `ied_responsible_unit`
  - 再联动 `ied_discipline_office`

## 14. 接口与页面之间的关联

这是另一个 AI 最需要理解的部分。

### 14.1 页面启动

1. 调 `GET /api/system/health`
2. 调 `GET /api/meta/form-schema`
3. 用 schema 初始化表单
4. 用 health 初始化系统状态显示

### 14.2 交付任务提交

1. 用户选择 `交付处理`
2. 用户拖入多个 `dwg`
3. 用户填写模块6前端字段
4. 前端本地执行 `required / required_when / format` 校验
5. 通过后调用 `POST /api/jobs/batch`
6. 后端返回 `batch_id + jobs[]`
7. 前端跳转到最近任务视图，并高亮本次 batch

### 14.3 纠错任务提交

1. 用户选择 `纠错`
2. 用户选择 `project_no`
3. 用户拖入一个或多个 `dwg`
4. 前端只做最小上传校验和必填校验
5. 调用词库任务接口
6. 后端返回多个独立任务
7. 前端跳转到最近任务视图，并以“纠错”标签高亮

### 14.4 翻版任务提交

1. 用户选择 `翻版`
2. 用户选择 `source_project_no`
3. 用户选择 `target_project_no`
4. 用户拖入一个或多个 `dwg`
5. 前端校验源项目号和目标项目号不同
6. 调用词库任务接口
7. 后端返回多个独立任务
8. 前端跳转到最近任务视图，并以“翻版”标签高亮

### 14.5 任务轮询

- 运行中的任务：
  - 每 2~3 秒轮询
- 空闲时：
  - 每 10~15 秒刷新

需要轮询的接口：

- `GET /api/jobs`
- 当前打开详情时额外调 `GET /api/jobs/{job_id}`

### 14.6 下载动作

#### 交付任务

- `package.zip` 按钮可点
- `IED计划.xlsx` 如果存在也可点

如果任务归档且 IED 已过期：

- `package.zip` 仍可下载
- `IED计划.xlsx` 按钮显示“已过期”

#### 纠错任务

- 只显示报告下载按钮
- 不显示 `package.zip`
- 不显示 `IED计划.xlsx`

#### 翻版任务

- 显示“替换后 DWG”下载按钮
- 显示报告下载按钮
- 不显示 `package.zip`
- 不显示 `IED计划.xlsx`

### 14.7 取消与重试

- 取消：
  - 对运行中任务允许点击
  - 点击后二次确认
  - 调 `POST /api/jobs/{job_id}/cancel`
- 重试：
  - 对失败/已取消任务允许点击
  - 调 `POST /api/jobs/{job_id}/retry`
  - 返回新任务后，前端跳转到新任务

## 16. 页面文案风格

- 中文为主
- 用语要像工程工具，不要像营销页面
- 状态文案建议：
  - `排队中`
  - `处理中`
  - `文档生成中`
  - `纠错检查中`
  - `翻版处理中`
  - `已完成`
  - `失败`
  - `已取消`
  - `取消请求已提交`
- 任务入口文案建议：
  - `交付处理`
  - `纠错`
  - `翻版`
- 错误文案不要只显示英文异常，前端应优先显示中文摘要，再允许展开原始错误。

## 17. 美观要求（给另一个 AI 的明确约束）

- 不要生成“白色大卡片 + 紫色主按钮 + 默认表格”的通用后台
- 页面要有明确主题
- 需要有：
  - 自定义配色变量
  - 自定义排版层级
  - 有节奏的留白
  - 上传区、任务列表、详情状态之间清晰视觉层级
- 允许使用轻微网格背景、工程线框、图纸边框式分割
- 三个任务入口卡片必须有明显差异，但不能破坏整体一致性
- 不允许做成花哨炫技风
- 首要目标是：专业、可信、长期可用

## 19. API 协作边界

- API 设计与最终实现由后端侧负责
- 前端 AI 负责：
  - `frontend/` 内页面、组件、状态管理、上传交互、轮询、下载 UI
  - 严格按本文档的接口契约做客户端
- 前端 AI 不负责：
  - 发明新的任务模型
  - 擅自改接口字段名
  - 擅自把 `纠错/翻版` 做成前端本地处理
  - 擅自改动后端文件路径或后端仓库结构

## 20. 给另一个 AI 的最终要求

如果把本文件交给另一个对话窗口，要求它：

1. 只在 `frontend/` 下工作
2. 严格按本文档的接口契约建前端
3. 不擅自改动后端接口语义
4. 不把 `required: false` 字段做成阻塞项
5. 不使用在线 CDN 资源

7. 做出一个美观、工程感强、可直接投入内网使用的页面
