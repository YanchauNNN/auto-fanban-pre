# 前后端最终落地任务纪要

更新时间：2026-03-13

## 1. 当前总目标

本仓库后续要从“模块化本地工具链 + fanban_m5 桌面小程序”继续演进为“内网前端 + 后端执行节点”的正式交付系统。

核心业务目标：

1. 用户通过前端上传一个或多个 DWG。
2. 后端在 Windows 执行节点上完成：
   - DWG -> DXF
   - 图框识别
   - 图签字段提取
   - A4 多页成组
   - 拆图 / 重命名
   - PDF + DWG 输出
   - 封面 / 目录 / 设计文件 / IED 计划生成
3. 用户在前端查看任务进度、下载产物、查看日志和失败原因。
4. 系统支持纠错（audit/check）和主交付流程并存。

## 2. 已经验证过、后续必须继承的经验

这些经验不是 fanban_m5 的局部技巧，而是正式后台必须吸收的运行能力：

### 2.1 CAD 环境探测与选择

1. AutoCAD 不能再写死路径。
2. 必须在执行节点启动时自动探测本机可用 CAD。
3. 版本应按高到低排序。
4. 当前已放开到支持 2010 及以上版本。
5. 后续正式后台应支持：
   - 展示可用 CAD 版本列表
   - 允许设置系统默认 CAD
   - 允许必要时按任务覆盖 CAD 版本

### 2.2 打印资源必须受管

1. `PC3 / PMP / CTB` 不能继续依赖用户机器上历史残留文件。
2. 之前已经出现过：
   - `monochrome.ctb` 被错误覆盖
   - `plot style` 丢失
   - `PMP` 解析失败
3. 当前 fanban_m5 已经验证出更安全的方向：
   - 程序受管打印资源
   - 打印前自动校验
   - 尽量不污染用户系统原始资源
4. 正式后台必须把“打印资源状态”纳入可视化管理。

### 2.3 启动与运行完整性检查

1. 之前桌面包在蓝屏后会出现 `_internal` 损坏，导致 `init.tcl` 报错。
2. 经验结论：
   - 运行前要做 bundle 完整性检查
   - 缺关键文件时要报“资源损坏/缺失”，不要直接抛底层 traceback
3. 正式后台同样需要：
   - AutoCAD 可用性检查
   - ODA 可用性检查
   - 打印资源检查
   - Office / 文档导出依赖检查

### 2.4 模块2~5 的关键算法结论

1. 图框检测不能继续用“首个命中图层即全局短路”的老逻辑。
2. 现在已经重构为：
   - 高优先图层先扫描
   - 未覆盖锚点再做低优先图层局部补检
3. A4 多页要支持：
   - Master/Slave 成组
   - 邻页补检
   - 误报告警修正
4. 文件命名规则当前已经固化为：
   - `外部编码 + 版次 + 状态 + 空格 + (内部编码)`

### 2.5 项目号自动推断

1. 现在已经有统一规则：
   - DWG 文件名开头前 4 位数字作为项目号候选
2. 这个能力不应只留在桌面 GUI。
3. 后续正式后台也必须保留，且以后端公共逻辑为准。

## 3. 当前仓库对“最终部署方式”的明确结论

### 3.1 是否必须使用 IIS

结论：IIS 不是必须，但正式部署时建议使用；不过 IIS 只建议承担“前端静态站点 + 反向代理”，不建议直接承载 CAD / Office 核心执行链。

推荐架构：

1. 前端：
   - React + Vite 构建后的静态文件
   - 可由 IIS 托管
2. 后端：
   - FastAPI + 本地任务队列 + worker
   - 独立运行，不要把 CAD/Office 执行直接塞进 IIS 应用池
3. 执行节点：
   - AutoCAD
   - ODA
   - Office
   - 打印资源
   - 本地 `storage/jobs`

### 3.2 前端里的 CAD / 打印设置到底控制什么

前端控制的不是浏览器用户自己电脑上的 CAD，而是“后端执行节点”上的 CAD 与打印资源。

后续应该做成：

1. 系统默认执行配置
   - 默认 CAD 版本
   - 默认 `PC3`
   - 默认 `CTB`
   - 默认 `PMP`
2. 任务创建时默认继承系统配置
3. 高级用户可按任务覆盖

## 4. 当前终端任务队列的真实执行方式

这个部分很重要，因为它直接决定后续多用户并发能力。

### 4.1 当前 API 运行时是单队列、单 worker

现状来自 `API/app/runtime.py`：

1. 运行时只有一个 `queue.Queue()`。
2. 只启动一个 `_worker_thread`。
3. `_worker_loop()` 串行从队列中取出 `job_id` 并执行。

因此当前真实行为是：

1. 多个用户同时上传 DWG，只会排队。
2. 一个用户同时发“主交付流程”和“纠错流程”，也会排队。
3. 当前并没有真正的多任务并行执行。

### 4.2 `max_workers / max_jobs / max_queue` 目前没有完整落地

`documents/参数规范_运行期.yaml` 和 `backend/src/config/runtime_config.py` 里虽然有：

1. `concurrency.max_workers`
2. `concurrency.max_jobs`
3. `concurrency.max_queue`

但当前 API runtime 并没有真正按这些值创建多 worker 或多进程执行器。

结论：

1. 这些配置目前更多是“规划值”。
2. 不是当前真实并发能力。

### 4.3 `max_parallel_dxf` 当前也没有真正生效成并发桥接

配置里有：

1. `module5_export.cad_runner.max_parallel_dxf`

但当前代码中只有配置定义，没有看到它真正驱动并发执行 source_dxf 的实现。

结论：

1. 当前模块5桥接不是“多 CAD 桥接并开”。
2. 当前更接近“单任务内按 source_dxf 串行处理”。

## 5. 当前能否同时打开多个 CAD 处理桥接

### 5.1 现在的真实答案

严格来说，当前正式 API 运行链并不具备“稳定支持多个 CAD 桥接并发”的能力。

原因：

1. 任务层面是单 worker 串行。
2. `max_parallel_dxf` 未真正落地为并发调度。
3. CAD 相关资源具有天然竞争风险：
   - AutoCAD 用户目录
   - Plotters / Plot Styles
   - 临时文件
   - accoreconsole 工作目录
   - .NET bridge / SCR / result.json

### 5.2 当前同一时刻可能发生的事

1. 当前桌面小程序单次通常只处理一个任务。
2. 当前 API 运行时也是串行取任务执行。
3. 所以现实中“同时打开多个桥接”的情况目前基本不会稳定发生。

### 5.3 后续如果要支持并发，必须先完成的前置条件

必须先做这些，才能谈多任务并发：

1. 任务执行器改成多 worker 模型
2. 每个任务使用完全独立的工作目录
3. CAD 桥接临时文件、SCR、task/result.json 全部隔离
4. 打印资源部署策略从“共享用户目录写入”升级为“受管、幂等、可重入”
5. 明确 AutoCAD 并发上限
   - 不建议一上来就无限并发
   - 更适合先做“1~2 个 CAD worker 槽位”
6. 明确主交付流程与纠错流程的资源竞争策略

## 6. 对后续正式前后端落实的推荐方向

### 6.1 第一步：先做“环境管理面板”

后续后台优先做的不是花哨任务页，而是下面这些：

1. 执行节点健康状态
2. CAD 探测结果
3. 当前默认 CAD
4. 打印资源状态
5. 关键依赖检查
6. 最近失败任务诊断

### 6.2 第二步：把任务执行和环境配置分层

推荐分成两套接口：

1. `system/runtime-config`
   - 管环境
2. `jobs/*`
   - 管任务

### 6.3 第三步：并发不要一步到位

推荐顺序：

1. 先保留单 worker，但把所有运行状态和资源隔离打透
2. 再扩成可配置的 2 个 worker
3. 最后再评估是否允许“交付流程 + 纠错流程”并行

## 7. 当前给未来实现保留的明确决策

1. IIS 可以用，但只用于前端静态站点与反向代理。
2. FastAPI + worker 要独立于 IIS 运行。
3. 前端中的 CAD / 打印设置控制的是执行节点，不是浏览器用户本机。
4. 后续后台必须继承 fanban_m5 已验证过的：
   - CAD 自动探测与版本选择
   - 打印资源受管
   - 启动完整性检查
   - 任务阶段化日志
   - 失败原因可视化
5. 当前队列模型还是单 worker 串行，后续并发要单独立项实现。
