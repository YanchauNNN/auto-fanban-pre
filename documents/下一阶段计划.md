# 下一阶段计划

更新时间：2026-03-22

## 1. 文档目的

本文档用于定义下一批次的三大新增能力：

- 工作量统计
- 自动归档
- 账号管理、权限控制与流程化工作量审批

目标不是只写概念，而是把这三块需求收敛成一份可以直接指导前后端联动开发的实施计划，并且保持与当前仓库架构一致。

本计划默认遵循以下约束：

- 配置参数、规则、路径模板、角色映射等“参数变量”继续落在 YAML 中。
- `documents_bin/姓名角色表.csv` 继续作为账号主数据基础。
- 高频运行态不强行塞进 YAML，而是继续沿用当前系统已经存在的任务 JSON 持久化方式；这部分属于“状态”，不是“参数”。
- 本批不引入独立数据库，优先复用现有 `storage/`、`Job/TaskGroup`、`manifest.json` 体系。
- 本批先把“初始工作量”和“流程时间记录”打通，最终折减算法后续再展开。

## 2. 当前代码现状判断

### 2.1 工作量统计现状

当前系统已经具备以下基础：

- DWG 解析后可以得到每张图纸的 `paper_variant_id` 和 `paper_size_text`。
- A4 多页成组已经能区分单帧和 `sheet_set`。
- 设计文件的 S 列图幅、IED 的工时字段、设计文件 Z 列工时字段都有清晰绑定。

当前系统还缺少以下能力：

- 没有“图幅折算为 A1 工作量”的现成服务。
- `work_hours` 目前是前端人工输入值，不是自动统计结果。
- 任务详情里的 `deliverable_outputs` 只返回数量、文件名、页数，没有返回图幅明细和 A1 折算结果。

结论：

- “设计文件 S 列”不是最佳统计源。
- 最快、最稳定、最简洁的路径应该直接使用流水线中的 `paper_variant_id` / `sheet_set page count` 计算工作量，再把结果写回任务详情和工作量模块。

### 2.2 自动归档现状

当前系统已经具备以下基础：

- 任务完成后能拿到 `package.zip`。
- 任务完成后能拿到 `IED计划.xlsx`。
- 原始上传 DWG 已经落在任务目录或任务组目录下。
- `engineering_no`、`subitem_no`、`album_internal_code`、`revision` 都已经存在可追溯来源。

当前系统还缺少以下能力：

- 没有归档按钮。
- 没有归档服务。
- 没有网络共享盘路径配置。
- 没有归档状态、归档时间、归档操作人、归档目标路径的持久化。

### 2.3 账号管理与权限现状

当前系统已经具备以下基础：

- 前端已经预留了“账号模块”和“工作量模块”入口。
- IED 编审链字段已经完整存在，且前端已按 `姓名@ID` 规则校验。
- `documents_bin/姓名角色表.csv` 已经给出了科室编码、科室、账号、姓名、角色、密码六列基表。

当前系统还缺少以下能力：

- 没有真实登录接口。
- 没有会话管理。
- 没有账号读取和密码修改服务。
- 没有基于账号角色的任务过滤。
- 没有任务发起者字段。
- 没有“室主任看本科室、设计人员看自己、所领导和管理员看全部”的后端权限控制。

### 2.4 工作量流程现状

当前系统已经具备以下基础：

- 设计人员、校核者、工种负责人、审核者、审定者对应的业务字段都已经存在：
  - `ied_prepared_by`
  - `ied_checked_by`
  - `ied_discipline_leader`
  - `ied_reviewed_by`
  - `ied_approved_by`
- 任务包详情页已经有下载区和后续动作区，适合新增 `归档`、`提交` 等操作按钮。

当前系统还缺少以下能力：

- 没有“提交后启动流程”的状态机。
- 没有审批节点的通过时间记录。
- 没有“待我处理”视图。
- 没有登录提醒或页面提醒。
- 没有绿色荧光跳动等当前节点强调样式。

## 3. 关键设计决策

### 3.1 任务载体怎么统一

方案 A：继续保留“单任务 + 任务包”双形态。

优点：

- 对现有接口改动较小。
- 历史任务兼容最直接。

缺点：

- 归档、提交、权限、工作量、审批全都要同时兼容两套对象。
- 前端卡片和详情页判断分支会继续增多。
- 后续商业化功能会越来越难维护。

方案 B：把所有新的上传任务统一提升为“任务包”，即使只有一个交付子任务也走 `TaskGroup`。

优点：

- 归档、工作量、流程、权限都统一挂在“任务包”上。
- UI 语义与用户理解一致，用户面对的都是“任务包”。
- 后续加驳回、补件、重归档、日志审计时更自然。

缺点：

- 需要改造当前 `create_batch` 的创建逻辑。
- 前端任务卡数据模型需要做一次统一梳理。

方案 C：新建独立的 `Submission` 模型，`Job` 和 `TaskGroup` 只作为底层执行对象。

优点：

- 语义最完整。

缺点：

- 本批开发量偏大。
- 不适合在没有数据库的前提下一次性引入。

推荐结论：

- 采用方案 B。
- 历史单任务数据保持只读兼容。
- 新创建的交付任务全部统一成任务包对象。

### 3.2 CSV 与密码修改怎么共存

方案 A：直接修改 `姓名角色表.csv` 中的密码列。

优点：

- 实现简单。

缺点：

- 明文密码持续存在，不适合商业化产品。
- 用户改密会污染主数据基表。
- CSV 更适合作为导入基础，不适合作为会频繁回写的凭据库。

方案 B：CSV 作为账号基表，系统把初始账号导入后，密码修改落到独立的认证状态文件中。

优点：

- 不破坏 CSV 作为基础台账的定位。
- 用户改密不会反复重写原始基表。
- 可以只在运行态文件里保存哈希值，不再保存明文。

缺点：

- 需要做一次账号同步与密码覆盖逻辑。

方案 C：本批直接引入数据库。

优点：

- 后续扩展最好。

缺点：

- 超出本批边界。

推荐结论：

- 采用方案 B。
- `documents_bin/姓名角色表.csv` 作为账号主数据来源。
- 初始密码仅用于首次导入。
- 实际认证使用哈希密码。
- 用户修改密码后只写入运行态认证文件，不回写 CSV。

### 3.3 工作量统计的计算来源怎么选

方案 A：读取最终生成的设计文件，直接拿 S 列图幅。

优点：

- 与用户看到的设计文件表面值一致。

缺点：

- 需要重新打开生成文档。
- 依赖下游产物，性能差。
- 一旦设计文件生成失败，工作量统计也会一起失效。

方案 B：直接使用流水线中的 `paper_variant_id` 和 `sheet_set` 页数据计算。

优点：

- 数据最早、最干净、最稳定。
- 不依赖 Office 产物二次解析。
- 适合作为后续工作量、归档、审批的统一底座。

缺点：

- 需要把统计结果额外暴露到任务详情和工作量模块。

方案 C：从 `manifest.json` 二次汇总。

优点：

- 比读取设计文件轻。

缺点：

- 仍然属于下游二次加工。
- `manifest.json` 当前还没带图幅明细，需要先扩充。

推荐结论：

- 采用方案 B。
- 设计文件 S 列继续作为下游展示结果。
- 后端以 `paper_variant_id` 为标准源头来计算 A1 折算值。

## 4. 总体架构推荐

推荐把本批改造拆成五个横向能力层：

### 4.1 账号与认证层

职责：

- 读取 CSV 用户基表。
- 提供登录、登出、当前用户、修改密码能力。
- 维护角色和科室信息。

建议新增目录：

- `backend/src/auth/`
- `API/app/routers/auth.py`
- `frontend/src/features/auth/`
- `frontend/src/features/account/`

### 4.2 任务归属与权限层

职责：

- 在任务创建时记录任务发起者快照。
- 在任务列表和任务详情接口中做服务端权限过滤。
- 为前端提供“当前用户是否可见、是否可操作”的显式字段。

建议新增或扩展对象：

- `TaskOwnerSnapshot`
- `RolePermissionPolicy`
- `VisibleScope`

### 4.3 工作量统计层

职责：

- 从 `frames` 和 `sheet_sets` 计算初始 A1 工作量。
- 记录图幅分布、图纸数、A1 折算总量。
- 向任务包详情、账号模块、工作量模块提供汇总结果。

建议新增目录：

- `backend/src/workload/`

### 4.4 流程审批层

职责：

- 根据 `AY / BB / BD / BF / BH` 建立审批流。
- 支持提交、流转、通过、时间记录。
- 向登录页、工作量模块、任务详情提供状态提示。

建议新增目录：

- `backend/src/workflow/`
- `API/app/routers/workflow.py`
- `frontend/src/features/workload/`

### 4.5 归档层

职责：

- 将任务包产物复制到指定归档路径。
- 自动拼装层级目录。
- 记录归档状态、归档目标路径、归档时间、归档操作人。

建议新增目录：

- `backend/src/archive/`
- `API/app/routers/archive.py`

## 6. 持久化与数据模型改造

### 6.1 推荐新增强类型模型

推荐不要把核心业务状态继续散落在 `metadata: dict[str, Any]` 中，而是把关键结构升成强类型字段。

建议新增：

- `UserIdentity`
- `TaskOwnerSnapshot`
- `WorkloadBreakdownItem`
- `WorkloadSummary`
- `ArchiveRecord`
- `WorkflowNodeState`
- `WorkflowState`

### 6.2 `TaskGroup` 推荐新增字段

建议新增：

- `owner: TaskOwnerSnapshot | None`
- `workload_summary: WorkloadSummary | None`
- `workflow_state: WorkflowState | None`
- `archive_record: ArchiveRecord | None`
- `visible_scope_hint: str | None`

原因：

- 任务包是用户视角中的真实业务对象。
- 工作量、归档、流程都应该挂在任务包上，而不是子任务上。

### 6.3 `Job` 推荐新增字段

建议新增：

- `owner: TaskOwnerSnapshot | None`
- `parent_task_package_id: str | None`

原因：

- 子任务详情也要知道是谁发起的。
- 审批和归档虽然挂在任务包上，但子任务详情页仍可能需要展示只读摘要。

### 6.4 认证状态推荐存放位置

建议新增：

- `storage/runtime/auth/password_overrides.json`
- `storage/runtime/auth/sessions.json`

原因：

- 密码变更和会话状态属于高频运行态。
- 不适合回写业务规范 YAML。

### 6.5 历史任务兼容策略

历史 `job.json / group.json` 没有发起者和流程信息，建议按以下策略兼容：

- 历史任务默认标记为 `legacy_unowned`。
- 设计人员默认不显示历史无归属任务。
- 管理员和所领导可以看到全部历史任务。
- 如果后续需要，可增加一次性回填脚本。

## 7. 工作量统计方案

### 7.1 计算源头

工作量统计不读取设计文件 S 列，直接使用流水线上下文：

- 单帧图纸使用 `frame.runtime.paper_variant_id`
- A4 成组图纸使用 `sheet_set.pages` 或 `generated_page_count`

### 7.2 计算规则

第一版推荐规则：

- `CNPE_A1 = 1.0`
- `CNPE_A0 = 2.0`
- `CNPE_A0+1/2 = 3.0`
- `CNPE_A2 = 0.5`
- `CNPE_A3 = 0.25`
- `CNPE_A4 = 0.125`
- `A1+1/1 = 2.0`
- `A1+1/2 = 1.5`
- `A1+1/4 = 1.25`
- 竖版图幅与同面积横版图幅折算值一致

第二版可逐步演进为：

- 默认按标准图幅面积相对 A1 面积计算
- YAML 允许少数特殊图幅做覆盖

### 7.3 输出结果

建议输出以下字段：

- `drawing_count`
- `single_frame_count`
- `sheet_set_count`
- `paper_variant_breakdown`
- `a1_equivalent_total`
- `initial_workload_a1`

其中：

- `a1_equivalent_total` 与 `initial_workload_a1` 第一阶段可以相同。
- 未来引入折减算法后，再增加 `final_workload_a1`。

### 7.4 推荐插入点

推荐在 `PipelineExecutor` 的交付主链中新增 `WorkloadService`：

- 时机：`A4 多页成组` 之后、`打包` 之前
- 产出：
  - 写入 `TaskGroup.workload_summary`
  - 写入 `manifest.json`
  - 写入任务详情接口序列化结果

### 7.5 前端展示

第一阶段展示位置：

- 任务包详情页下载区附近显示“初始工作量：X A1”
- 最近任务卡显示“图纸数 / A1 折算”
- 账号模块显示个人累计工作量
- 工作量模块显示按任务维度的工作量卡片

## 8. 自动归档方案

### 8.1 触发方式

归档按钮新增位置：

- 任务包详情页“聚合下载”区域
- 子任务详情页如果属于交付任务，也可展示只读归档状态和跳转入口

点击后触发后台归档动作。

### 8.2 归档内容

第一阶段固定归档以下内容：

- `package.zip`
- `IED计划.xlsx`
- 原始上传 DWG

覆盖规则：

- 如果目标位置已有同名文件，则覆盖。

### 8.3 目标目录结构

按你给出的业务规则，第一阶段目录层级定义为：

`归档根路径 / engineering_no / subitem_no / album_internal_code / revision`

示例：

```text
E:\project\auto-fanban-pre\tmp\1818\NE\18185NE-JGS11\A
```

其中：

- `engineering_no` 来自 001 图纸或前端合并参数
- `subitem_no` 来自 001 图纸或前端合并参数
- `album_internal_code` 来自文档派生值
- `revision` 使用统一卷册版次

### 8.4 归档执行策略

推荐不要在请求线程里直接长时间复制文件，建议新增轻量动作执行器。

建议特性：

- 不占用 CAD slot
- 有开始时间、结束时间、状态、错误信息
- 失败后可重试

### 8.5 归档记录

建议记录：

- `status`
- `target_root`
- `target_dir`
- `archived_files`
- `operator_account`
- `operator_name`
- `started_at`
- `finished_at`
- `error`

### 8.6 管理员能力

管理员账号模块中增加：

- 查看当前归档根路径
- 修改归档根路径
- 测试路径连通性
- 查看最近归档记录

## 9. 账号管理与权限方案

### 9.1 账号基表

继续使用：

- `documents_bin/姓名角色表.csv`

字段含义：

- 科室编码
- 科室
- 账号
- 姓名
- 角色
- 密码

第一阶段以“账号”为主键，避免姓名重复带来的歧义。

### 9.2 登录与会话

推荐能力：

- 登录
- 登出
- 获取当前用户
- 修改密码
- 会话过期

推荐方式：

- HttpOnly Cookie + 服务端会话状态

原因：

- 适合内网 Web 应用。
- 不必在前端长期持有令牌。
- 比完全无状态 JWT 更容易做强制失效和改密后踢出。

### 9.3 角色映射

第一阶段统一角色枚举：

- `design`
- `office_director`
- `institute_leader`
- `admin`

### 9.4 权限规则

第一阶段任务可见性规则：

- 设计人员：仅自己发起的任务
- 室主任：本室全部任务
- 所领导：全部任务
- 管理员：全部任务

第一阶段界面差异：

- 非设计人员在任务卡上显示任务发起者姓名
- 设计人员不额外显示发起者姓名

### 9.5 任务发起者快照

任务创建时必须写入：

- 发起者账号
- 发起者姓名
- 发起者角色
- 发起者科室编码
- 发起者科室名称

这部分必须在创建任务包时就固化，不能后面再靠 CSV 反查补齐。

### 9.6 账号模块页面

第一阶段包含：

- 当前账号信息
- 角色
- 科室
- 修改密码
- 我的工作量摘要
- 我的任务统计

管理员后续扩展：

- 查看账号目录同步结果
- 查看异常账号行
- 查看最近登录记录

## 10. 流程化工作量模块方案

### 10.1 流程载体

流程以“任务包”为载体，不以单张图或子任务为载体。

### 10.2 启动方式

设计人员在任务包详情页点击 `提交` 后启动流程。

启动条件：

- 交付任务成功
- 已生成必要产物
- 编审链相关账号都能映射到真实用户

### 10.3 节点定义

第一阶段节点固定为：

1. 编制者
2. 校核者
3. 工种负责人
4. 审核者
5. 审定者

### 10.4 节点动作

第一阶段只做：

- `提交`
- `通过`

但数据结构预留：

- `退回`
- `备注`
- `加签`

### 10.5 时间记录

每个节点都要记录：

- `activated_at`
- `acted_at`
- `actor_account`
- `actor_name`
- `decision`

这些时间将作为后续工作量折减算法的输入，不在本批做最终扣减计算。

### 10.6 提醒方式

第一阶段只做站内提醒：

- 登录后顶部提醒
- 工作量模块中的待办卡片
- 当前轮到自己时任务卡绿色荧光跳动

第一阶段不做：

- 邮件
- 企业微信
- 短信

### 10.7 工作量模块页面

建议拆成三个视图：

- `待我处理`
- `我发起的流程`
- `我的工作量`

每张卡片建议展示：

- 图册号
- 项目号
- 发起者
- 当前节点
- 初始工作量
- 当前状态
- 提交时间
- 当前节点激活时间
- 通过按钮

## 12. 推荐实施顺序

### 阶段 1：基础骨架与统一任务包

目标：

- 把所有新上传任务统一成任务包
- 为任务包加入发起者快照和业务扩展字段
- 加入管理员可写的运行期 YAML 覆盖机制

交付标准：

- 新任务都以任务包形式显示
- 旧任务仍可只读查看

### 阶段 2：账号与认证

目标：

- 接入 CSV 账号目录
- 实现登录、登出、当前用户、改密
- 实现角色枚举和服务端会话

交付标准：

- 未登录不能访问业务工作台
- 登录后能拿到当前用户信息
- 修改密码后重新登录生效

### 阶段 3：权限过滤

目标：

- 后端按角色过滤 `/api/jobs`
- 任务详情做权限校验
- 最近任务卡按身份展示不同信息

交付标准：

- 设计人员看不到别人的任务
- 室主任能看到本科室任务
- 所领导和管理员能看全部任务

### 阶段 4：工作量统计

目标：

- 增加 A1 折算服务
- 任务包详情展示初始工作量
- 工作量模块出现第一版任务卡视图

交付标准：

- 每个成功交付任务包都有 `initial_workload_a1`
- 工作量结果与图幅明细可追溯

### 阶段 5：流程化审批

目标：

- 任务包详情增加 `提交`
- 工作量模块支持 `通过`
- 节点时间完整记录

交付标准：

- 流程能按五个节点顺序流转
- 当前待处理人能收到页面提醒

### 阶段 6：自动归档

目标：

- 任务包详情增加 `归档`
- 支持管理员配置归档根路径
- 支持覆盖式复制

交付标准：

- 指定归档路径下能生成正确层级目录
- `package.zip`、IED、原始 DWG 全部可落位

### 阶段 7：产品化补强

目标：

- 完善状态提示
- 完善错误文案
- 完善前后端测试
- 补齐运维文档

## 14. 本批默认不做的内容

## 15. 面向商业化的延申建议

如果目标是逐步走向成熟商业化产品，建议在本批完成后按顺序继续发散：

- 增加审批退回、补充说明、再次提交能力。
- 增加任务操作审计日志，记录谁在什么时候下载、归档、提交、通过。
- 增加归档失败重试队列和归档健康检查。
- 增加账号锁定、密码复杂度、首次登录强制改密。
- 把 `姓名@ID` 文本输入升级为真实的账号选择器，杜绝手工拼写错误。
- 增加管理员工作台，查看账号同步、待审批积压、归档异常、工作量统计报表。
- 增加部门维度和个人维度的导出报表。
- 当任务量继续增大时，再引入数据库，把认证、流程、归档、统计从 JSON 迁移出去。

## 16. 本计划的推荐落地结论

本批最值得坚持的三条主线是：

1. 不从设计文件反读工作量，直接使用流水线上游的标准图幅结果。
2. 不把登录、归档、流程继续做成前端拼接能力，而是升级为后端一等业务模型。
3. 不继续让“单任务”和“任务包”并存为未来主结构，而是把所有新上传统一成任务包。

如果按这个方向推进，这一批不仅能满足你当前提出的三大业务基线，也能为后面真正做成产品级网页打下比较稳的底座。
