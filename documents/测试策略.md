# 测试策略

## 目的

- 当前仓库已经不适合每次改动都无差别跑完整测试。
- 日常回归应优先保证反馈速度，部署/环境探测类测试应独立运行。
- 本策略的目标是：保留测试覆盖，不靠削弱断言换速度。

## 现状结论

- 默认日常单测的主要瓶颈不是数量，而是环境探测测试。
- [test_probe_target_env.py](E:/project/auto-fanban-pre/backend/tests/integration/test_probe_target_env.py:1) 会实际调用 [probe_target_env.ps1](E:/project/auto-fanban-pre/tools/probe_target_env.ps1:1)，并触发 Word/Excel COM、PDF 导出、模板可读性检查。
- 这类测试属于集成/环境级验证，不应与纯单元测试混跑。

## 分层原则

- `unit`
  - 纯 Python 逻辑、模型、参数校验、任务序列化、路由行为。
  - 要求快、稳定、可在开发机频繁执行。
- `integration`
  - 依赖 PowerShell、Office COM、AutoCAD 探测、真实文件系统环境。
  - 单独跑，不纳入日常最小回归。
- `slow`
  - 即使在本机可执行，也明显重于常规单元测试。
  - 应作为补充验证，而不是默认每次提交都跑。

## 推荐阶段

### 1. quick

- 用途：日常改动后的默认回归。
- 命令：

```powershell
powershell -File tools/run_test_stage.ps1 -Stage quick
```

- 覆盖范围：
  - `backend/tests/unit`
  - 自动排除 `slow` 和 `integration`

### 2. cad

- 命令：

- 主要覆盖：

  - `test_cad_dxf_executor.py`
  - `test_frame_detector.py`

- 用途：环境探测脚本或部署相关改动。

### 6. all

- 用途：合并前或重要里程碑前的完整验证。
- 命令：

```powershell
powershell -File tools/run_test_stage.ps1 -Stage all
```

## 使用建议

- 只改 API：跑 `quick + api`
- 只改模块5：跑 `quick + cad`
- 只改模块6：跑 `quick + docs`
- 改部署/环境探测：跑 `quick + env`
- 准备提交大改动：跑 `quick + 对应模块阶段`
- 准备合并里程碑：跑 `all`

## 当前仓库判断

- 当前大多数测试不需要简化。
- 真正需要的是：
  - 把重型测试从日常回归中剥离
  - 建立阶段化运行习惯
  - 明确不同改动对应的最小验证集
