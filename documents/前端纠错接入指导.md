# 前端纠错接入指导

## 1. 当前状态

- 后端 `纠错` 接口已经完成，接口代码在 `API/` 和 `backend/src/audit_check/`。
- 当前主工作区 **还没有正式落地的 `frontend/` 工程**。
- `.worktrees/frontend-blueprint/frontend` 里已经有前端蓝图和可运行原型，但它还不是主工作区最终交付物。
- 当前不需要做前端静态托管，前端工程师先用本地 dev server 对接 API 即可。

结论：

1. **当前还没有接入正式前端。**
2. **UI 层烟测应在前端完成接入后执行。**
3. 后端 API 现在已经具备前端接入条件。

---

## 2. 本轮接入范围

这次前端只接 `纠错`，不要同时推进 `翻版`。

目标是让用户在前端点击 `纠错` 后可以完成：

1. 选择或输入 `project_no`
2. 上传一个或多个 `dwg`
3. 创建纠错任务
4. 轮询任务状态
5. 任务完成后弹出摘要弹窗
6. 在任务详情中下载 `report.xlsx`

本轮不要做：

- `翻版` 真正提交
- 前端本地词库解析
- 前端本地文本扫描
- 静态托管
- 登录权限

---

## 3. 后端接口契约

### 3.1 健康检查

`GET /api/system/health`

用途：

- 页面启动时检查 API 是否在线
- 顶部状态条展示系统可用性

最少使用字段：

- `status`
- `ready`
- `storage_writable`
- `worker_alive`
- `autocad_ready`
- `office_ready`

---

### 3.2 获取纠错表单元数据

`GET /api/meta/form-schema`

前端本轮对 `纠错` 只需要读：

```json
{
  "audit_replace": {
    "project_options": ["2016", "1418", "1818", "..."]
  }
}
```

用途：

- 生成 `project_no` 下拉选项
- 给前端做推荐项和快速选择

注意：

- 本轮 `纠错` 前端 **不要** 去消费 `deliverable.sections`
- `纠错` 只需要 `audit_replace.project_options`

---

### 3.3 创建纠错任务

`POST /api/jobs/audit-replace`

`Content-Type: multipart/form-data`

字段：

- `mode`
  - 固定传：`check`
- `params_json`
  - JSON 字符串
  - 本轮最少包含：
    - `project_no`
- `files[]`
  - 一个或多个 `.dwg`

请求示例：

```http
POST /api/jobs/audit-replace
Content-Type: multipart/form-data

mode=check
params_json={"project_no":"2026"}
files[]=20261NH-JGS51-B合并版.dwg
```

### 3.4 创建成功响应

后端返回 `201 Created`

示例结构：

```json
{
  "batch_id": "batch-20260313153000-a1b2c3",
  "jobs": [
    {
      "job_id": "job-xxx",

      "source_filename": "20261NH-JGS51-B合并版.dwg",
      "task_kind": "audit_check",

      "project_no": "2026",
      "status": "queued",
      "stage": "INIT",
      "percent": 0,
      "message": "",
      "created_at": "2026-03-13T15:30:00+08:00",
      "finished_at": null,
      "artifacts": {
        "package_available": false,

        "replaced_dwg_available": false
      },

    }
  ]

```

---

### 3.5 参数错误响应

当 `project_no` 留空且无法从文件名推断时，后端返回 `422`

示例：

```json
{
  "detail": {
    "upload_errors": {},
    "param_errors": {
      "project_no": ["required_for_audit_check"]
    }
  }
}
```

前端处理要求：

- 把这个错误直接展示在 `project_no` 字段下方
- 不要把它包装成通用“提交失败”

注意：

- 后端支持“文件名推断 `project_no`”
- 但前端本轮烟测时 **仍建议显式提交 `project_no`**
- 这样更稳定，也更方便排查问题

---

### 3.6 任务列表

`GET /api/jobs`

前端对 `纠错` 任务至少需要使用：

- `job_id`
- `source_filename`
- `task_kind`
- `job_mode`
- `status`
- `stage`
- `percent`
- `message`
- `created_at`
- `finished_at`
- `findings_count`
- `affected_drawings_count`
- `artifacts.report_available`

筛选规则：

- 本轮至少要能把 `task_kind === "audit_check"` 的任务在 UI 上区分出来

---

### 3.7 任务详情

`GET /api/jobs/{job_id}`

前端至少使用：

- `job_id`
- `source_filename`
- `task_kind`
- `job_mode`
- `project_no`
- `status`
- `stage`
- `percent`
- `message`
- `current_file`
- `flags`
- `errors`
- `findings_count`
- `affected_drawings_count`
- `top_wrong_texts`
- `top_internal_codes`
- `artifacts.report_available`
- `artifacts.report_download_url`

### 3.8 任务详情中纠错摘要字段

当前后端已经直接返回：

- `findings_count`
- `affected_drawings_count`
- `top_wrong_texts`
- `top_internal_codes`

这四个字段就是弹窗摘要的主要数据源。

---

### 3.9 下载报告

`GET /api/jobs/{job_id}/download/report`

返回：

- `report.xlsx`

前端要求：

- 仅当 `artifacts.report_available === true` 时显示下载按钮
- 文件名直接使用后端响应文件名

## 4. 前端接入要求

### 4.1 页面入口

在现有三入口设计里，把 `纠错` 接成真实可用入口。

用户操作路径建议为：

1. 首页点击 `纠错`
2. 打开 `纠错配置` 弹窗
3. 选择 `project_no`
4. 选择一个或多个 `dwg`
5. 点击 `创建任务`
6. 返回任务列表轮询
7. 任务完成后弹出结果摘要

---

### 4.2 表单要求

本轮 `纠错` 表单只保留：

- `project_no`
- `dwg_files[]`

不要混入：

- 交付处理的模块6参数
- `source_project_no`
- `target_project_no`

推荐交互：

- `project_no` 用可搜索下拉
- 默认不为空时直接提交
- 允许用户手动清空，但清空后要提示“将依赖文件名推断”

---

### 4.3 提交逻辑

前端提交时，统一走：

- `mode=check`
- `params_json={"project_no":"用户当前选择值"}`
- `files[]`

建议：

- 烟测时不要依赖后端推断，直接显式传 `project_no`
- 后续如要支持“留空自动推断”，可以作为增强，不影响本轮接入

---

### 4.4 列表展示

任务列表中 `纠错` 任务应明显区别于 `交付处理`

至少区分：

- 类型标签：`纠错`
- 状态标签：`queued / running / succeeded / failed`
- 结果摘要：
  - `错误数`
  - `受影响图纸数`

不要显示：

- `package.zip`
- `IED计划.xlsx`

---

### 4.5 详情页展示

`纠错` 任务详情页至少包含：

- 源 DWG 文件名
- 项目号
- 任务状态
- 当前阶段
- 错误数
- 受影响图纸数
- `top_wrong_texts`
- `top_internal_codes`
- 错误信息
- `下载报告`

不要显示：

- `package.zip`
- `IED计划.xlsx`

---

### 4.6 摘要弹窗

当任务从 `running` 变为 `succeeded` 且 `findings_count > 0` 时，自动弹出摘要窗。

弹窗最少展示：

- 总错误数：`findings_count`
- 受影响图纸数：`affected_drawings_count`
- 前 10 个错误文本：`top_wrong_texts`
- 前 10 个受影响内部编码：`top_internal_codes`
- 按钮：
  - `下载完整报告`
  - `关闭`

如果 `findings_count == 0`：

- 不弹错误摘要窗
- 只给一个轻量成功提示

8. 详情页能显示：
   - `findings_count`
   - `affected_drawings_count`
   - `top_wrong_texts`
   - `top_internal_codes`
9. 能点击下载 `report.xlsx`

### 5.4 烟测记录要求

前端工程师执行完烟测后，至少反馈：

- 提交是否成功
- 任务是否进入 `succeeded`
- 是否弹出摘要窗

- `report.xlsx` 是否能成功下载
- 若失败，提供：
  - 前端控制台报错
  - 网络请求截图或文本
