## 2. 系统总体定位

这是一个运行在 Windows 执行节点上的 DWG 批处理系统，服务对象是内网用户。整体形态是：

- 前端：浏览器工作台
- API：FastAPI
- 后端核心：Python 流水线编排
- CAD 执行：ODA + AutoCAD/AcCoreConsole + .NET Bridge + 少量 LISP 兜底
- 文档执行：Word/Excel 模板 + Office COM 导出 PDF
- 部署形态：离线终端部署包，可通过 IIS 提供静态前端和 `/api` 反代

当前系统支持两条主业务链：

### 2.1 流水线 A：交付处理

输入：

- 一个或多个 `dwg`
- 项目号与文档生成参数

输出：

- `package.zip`
- `IED计划.xlsx`
- 任务详情中的结构化结果摘要

处理主链：

`DWG -> DXF -> 图框检测 -> 图签提取 -> A4 多页成组 -> 标题栏一致性修复 -> 拆分/打印 -> 生成封面/目录/设计文件/IED -> 打包`

### 2.2 流水线 B：纠错 / 翻版

当前已正式可用的是：

- `audit_check` / `mode=check`：词库扫描，只出报告，不改图

当前已有规划与部分基础设施，但尚未完全落地的是：

- `audit_replace` / `mode=replace`：翻版替换，目标是生成替换后的 DWG，并可串联交付流程

与这条链强相关的设计文档在：

- `documents/翻版纠错方案.md`
- `documents/纠错方案.md`

## 4. 源事实与配置优先级

维护这个仓库时，最重要的原则是“不要在代码里重新发明业务规则”。

### 4.1 业务源事实

`documents/参数规范.yaml`

这里定义了：

- 前端表单字段与 `required_when`
- 封面/目录/设计文件/IED 的模板选择与落点
- 图框检测参数
- 图签 ROI 与字段定义
- A4 多页规则

### 4.2 运行源事实

`documents/参数规范_运行期.yaml`

这里定义了：

- 存储目录
- ODA / AutoCAD / PDF 引擎路径或模式
- `module5_export`
- `audit_check`
- `deliverable_consistency_fix`
- 上传限制
- 生命周期策略
- 日志与并发配置

### 4.3 环境变量覆盖

通过 `FANBAN_*` 环境变量可覆盖运行期配置。

典型例子：

- `FANBAN_STORAGE_DIR`
- `FANBAN_SPEC_PATH`
- `FANBAN_RUNTIME_SPEC_PATH`
- `FANBAN_UPLOAD_LIMITS__MAX_FILES`

实现位置：

- `backend/src/config/spec_loader.py`
- `backend/src/config/runtime_config.py`

### 4.4 资源源事实

打印和锚点相关资源主要来自：

- `documents/Resources/`

其中包括：

- `打印PDF2.pc3`

## 5. 后端运行时总控

后端真实总控在 `API/app/runtime.py` 的 `DeliverableApiRuntime`。

它负责装配下面这些组件：

- `JobManager`
- `GroupManager`
- `DocParamValidator`
- `FormMetadataService`
- `SharedPrepService`
- `CADSlotPool`
- 工作队列与后台 worker

### 5.1 当前真实执行模型

当前线上实现仍然偏保守：

- 任务进入队列
- 后台线程处理
- 交付与纠错通过统一运行时管理

虽然运行期配置里已经有并发参数，且 `CADSlotPool` 已落地，但“完整的多队列、多 worker、资源调度器”还没有完全放开。这个状态在下列文档里有说明：

- `documents/后端并发执行与 CAD 资源池改造方案.md`
- `documents/archieve/前后端最终落地任务纪要.md`

### 5.2 任务类型

当前主要有：

- `deliverable`
- `audit_replace`
  - 当前真正跑的是 `mode=check`
  - 业务上会展示成 `audit_check`

### 5.3 Group / Child 机制

当用户在交付任务中勾选“同时跑纠错”时，系统不是串行提交两个独立任务，而是创建一个 group：

- parent：`TaskGroup`
- child 1：`deliverable`
- child 2：`audit_check`

两个 child 可共享一次前处理结果。

相关实现：

- `backend/src/models/task_group.py`
- `backend/src/pipeline/group_manager.py`
- `backend/src/pipeline/shared_prep.py`
- `API/app/routers/jobs.py`

## 6. 交付流水线 A 详细结构

主执行器是 `backend/src/pipeline/executor.py` 中的 `PipelineExecutor`。

### 6.1 阶段定义

系统把交付流程拆成一系列阶段，便于：

- 进度显示
- 局部跳过
- flags 汇总
- 失败定位

关键阶段包括：

1. `INGEST`
2. `CONVERT_DWG_TO_DXF`
3. `DETECT_FRAMES`
4. `VERIFY_FRAMES_BY_ANCHOR`
5. `SCALE_FIT_AND_CHECK`
6. `EXTRACT_TITLEBLOCK_FIELDS`
7. `A4_MULTIPAGE_GROUPING`
8. `FIX_TITLEBLOCK_CONSISTENCY`
9. `SPLIT_AND_RENAME`
10. `EXPORT_PDF_AND_DWG`
11. `GENERATE_DOCS`
12. `PACKAGE_ZIP`

### 6.2 前处理共享

当是 group 模式时，交付子任务和纠错子任务会共享一套前处理产物：

- DXF
- frames
- sheet_sets
- titleblock_extracts
- prep_summary

这些数据会落到 `shared` 目录并由 `SharedPrepService` 管理。

### 6.3 文档上下文构建

`PipelineExecutor` 会把下面几类数据汇总成 `DocContext`：

- 前端参数
- 001 图纸提取值
- 派生字段
- 单页图框
- A4 成组
- 规则与映射

这是文档生成链唯一应该消费的核心上下文，不应该让文档生成模块重新读 DWG 或 DXF。

## 7. 纠错流水线 B 详细结构

主执行器是：

- `backend/src/audit_check/executor.py`

当前正式能力是 `check` 模式。

### 7.1 处理步骤

大体流程是：

1. 复用共享前处理，或独立跑一遍前处理
2. 加载词库 Excel
3. 通过 .NET Bridge 扫描 DWG/DXF 内的文本实体
4. 给扫描文本补上 ROI / internal_code / frame 上下文
5. 用匹配引擎判断是否命中“其他项目词条”
6. 生成 `report.json` 与 `report.xlsx`

### 7.2 核心组件

- `lexicon.py`
  - 词库加载
- `matcher.py`
  - 命中与误报抑制
- `roi_mapper.py`
  - ROI/图纸上下文标注
- `bridge.py`
  - 调用 CAD 侧扫描
- `reporting.py`
  - 报告输出

### 7.3 误报抑制

当前已经明确写入规格的误报抑制包括：

- 日期/时间上下文
- 编号内嵌
- 边界匹配
- 动态字段跳过
- XREF 不递归改写

业务规格详见：

- `documents/翻版纠错方案.md`

这一层是整个系统的“公共语言”。

### 8.1 `Job`

文件：

- `backend/src/models/job.py`

作用：

- 描述一次任务执行

典型字段：

- `job_id`
- `job_type`
- `project_no`
- `status`
- `progress`
- `artifacts`
- `input_files`
- `params`
- `options`

### 8.2 `FrameMeta`

文件：

- `backend/src/models/frame.py`

这是单个图框的完整表示，分两部分：

- `runtime`
  - 外框 bbox
  - 图幅
  - 比例
  - 来源文件
  - 产物路径
  - flags
- `titleblock`
  - 内外部编码
  - 专业
  - 比例文字
  - 标题
  - 页码
  - 状态/版次/日期

### 8.3 `SheetSet`

文件：

- `backend/src/models/sheet_set.py`

用于表达 A4 多页说明图的成组结果。

关键点：

- 有 `master_page`
- 有 `pages`
- 有 `page_total`
- 有继承图签逻辑
- 有一致性校验 flags

### 8.4 `DocContext`

文件：

- `backend/src/models/doc_context.py`

这是文档生成总输入。

包含：

- `params`
- `derived`
- `frames`
- `sheet_sets`
- `rules`

文档链基本都应围绕它工作。

### 8.5 `TaskGroup`

文件：

- `backend/src/models/task_group.py`

用于表示一个父级任务包，当前最主要的用途是：

- 交付 + 纠错组合提交

## 9. 图框检测与图签提取链

这一段是维护成本最高、也最容易出真实样本问题的部分。

### 9.1 图框检测入口

文件：

- `backend/src/cad/frame_detector.py`

当前主要模式是“几何优先 + 锚点校验/补检”的组合。

### 9.2 候选矩形

文件：

- `backend/src/cad/detection/candidate_finder.py`

主要逻辑：

- 优先闭合 `LWPOLYLINE` / `POLYLINE`
- 再做 `LINE` 重建
- 支持按 layer 优先级扫描
- 支持局部窗口补检

### 9.3 纸张拟合

文件：

- `backend/src/cad/detection/paper_fitter.py`

作用：

- 把候选矩形拟合到标准图幅
- 得到 `paper_variant_id`
- 计算 `sx/sy`
- 选择 ROI profile

### 9.4 锚点策略

文件：

- `backend/src/cad/detection/anchor_first_locator.py`
- `backend/src/cad/detection/anchor_calibrated_locator.py`

当前主路径是 `anchor_first_locator.py`。

它的关键思路：

1. 扫描锚点文本
2. 先做全局 layer 内匹配
3. 未匹配项再做局部补检
4. 必要时做动态 layer 补检
5. 对 A4 做同簇扩展

这解释了为什么很多“图框没识别”问题，本质不是 ROI 坏了，而是 layer 覆盖不对。

### 9.5 实际维护案例：图框层未覆盖

在真实样本 `test/xuexiaohang-feedback/20261NH-JGS51.dwg` 中，发生过这样的问题：

- 锚点文本有 32 处
- 当前正式配置只稳定配到 14 个图框
- 原因不是锚点错位，而是大量真实图框在 `TSSDPRO_book` layer
- 当前图框扫描配置没有覆盖这一层

这类问题的正确排查顺序应该是：

1. 看检测统计里锚点数和图框数是否明显失衡
2. 看当前 spec 中的 `layer_order` / `local_only_layers`
3. 直接检查漏页图框实际所在 layer
4. 再判断是否需要改算法，而不是先怀疑 ROI

### 9.6 图签提取

文件：

- `backend/src/cad/titleblock_extractor.py`

作用：

- 通过 ROI 从 DXF 文本中提取图签字段
- 支持缓存同一 DXF 的文本扫描
- 支持 A4 页码标记解析
- 记录 `raw_extracts`
- 检查比例不一致

这是后续文档生成、A4 继承、标题栏一致性修复的基础。

## 10. A4 多页成组

文件：

- `backend/src/cad/a4_multipage.py`

相关说明文档：

- `documents/archieve/A4特殊逻辑.md`

### 10.1 目标

识别“001 多页说明图”这类场景：

- 只有第一页有完整图签
- 后续页只有页码或少量说明

### 10.2 主要动作

- 判断哪些 frame 属于 A4 / SMALL5
- 按空间距离构建 cluster
- 识别 master page
- 识别 slave 页码
- 继承图签元数据
- 做一致性检查

### 10.3 产物

输出 `SheetSet`，后续拆图和文档层都按“一个多页文件”看待它。

## 11. 拆图、打印与 AutoCAD 主路径

这是当前系统最复杂、最具工程性的部分。

### 11.1 历史演进

仓库历史上经历过两个阶段：

1. 纯 Python 裁切/打印链
2. AutoCAD / AcCoreConsole / .NET Bridge 主路径

当前主干已经明显偏向第二种。

最值得读的历史总结：

- `documents/archieve/模块5工作总结-旧技术管线.md`
- `documents/archieve/模块5_CAD路径工作总结.md`

### 11.2 Python 侧 orchestrator

文件：

- `backend/src/cad/cad_dxf_executor.py`

它不是直接做 CAD 图形处理，而是：

- 按 source DXF 分组
- 准备 task.json
- 准备输出目录
- 确保打印资源就绪
- 调用 `AcCoreConsoleRunner`
- 读取 result.json
- 回填 `FrameMeta` / `SheetSet`

### 11.3 AutoCAD 进程启动

文件：

- `backend/src/cad/accoreconsole_runner.py`
- `backend/src/cad/autocad_path_resolver.py`

职责：

- 找到 `accoreconsole.exe`
- 组装命令行
- 传入脚本与 profile
- 记录 trace / result / log

### 11.4 槽位资源池

文件：

- `backend/src/cad/slot_pool.py`

它为每个 CAD 槽位准备隔离运行环境：

- profile
- Plotters
- Plot Styles
- PMP
- temp

- logs

这是当前并发设计的基础设施，但还不是完全放开的并发调度器。

### 11.5 打印资源管理

文件：

- `backend/src/cad/plot_resource_manager.py`

维护原则：

- 程序受管资源优先
- 不依赖用户历史 profile
- 启动前确保 `PC3 / CTB / PMP` 齐全

### 11.6 CAD 侧 Bridge

目录：

- `backend/src/cad/dotnet/Module5CadBridge/`

主要入口：

- `Commands.cs`

- `PlotEngine.cs`

- `split_only`

### 11.7 LISP 脚本

文件：

- `backend/src/cad/scripts/module5_bootstrap.scr`

## 12. 标题栏一致性修复

文件：

- `backend/src/cad/titleblock_consistency.py`
- `backend/src/cad/titleblock_consistency_bridge.py`
- `.NET` 侧 `TitleblockConsistencyFixer.cs`

这是当前流水线里很容易被忽视、但对产物稳定性很重要的一层。

作用：

- 检查纸张说明、比例说明、A4 页码修订等文本是否与系统推导值一致
- 如不一致，生成 patch plan
- 通过 CAD 侧精确改字并另存修正文件

这套修复逻辑是“图签提取之后、拆图之前”的重要补偿层。

## 13. 文档生成链

文档生成不是附属功能，而是交付流程后半段的核心产物链。

### 13.1 派生规则

文件：

- `backend/src/doc_gen/derivation.py`

职责：

- 从 `DocContext` 计算封面/目录/图册编码
- 计算 `cover_internal_code`、`catalog_internal_code`
- 生成中英文标题
- 计算统一文档版次
- 计算专业英文、设计阶段英文等

### 13.2 封面

文件：

- `backend/src/doc_gen/cover.py`

逻辑特点：

- 按项目号与封面 variant 选择模板
- 通过绑定把值写入 Word / OLE
- 调用 PDF 导出器输出 PDF

### 13.3 目录

文件：

- `backend/src/doc_gen/catalog.py`

逻辑特点：

- 行顺序固定：封面 -> 目录 -> 图纸
- 支持 1818 与非 1818 差异
- 负责目录页数回填
- 升版标记只影响目录中的标记列，不参与版次计算

### 13.4 设计文件

文件：

- `backend/src/doc_gen/design.py`

逻辑特点：

- 只生成 Excel，不导 PDF
- 包含封面/目录/图纸三类行
- 使用 001 或 sheet-set 继承值补齐

### 13.5 IED

文件：

- `backend/src/doc_gen/ied.py`

逻辑特点：

- 只生成 Excel
- 不进入 `package.zip`
- 依赖大量 IED 字段校验

### 13.6 PDF 引擎

文件：

- `backend/src/doc_gen/pdf_engine.py`

当前优先使用：

- Office COM

可以按运行期配置切换或兜底，但整体思路仍然是优先保证产物与模板一致性。

### 13.7 参数校验

文件：

- `backend/src/doc_gen/param_validator.py`

职责：

- 校验前端提交字段
- 处理 `required_when`
- 校验日期、人名@ID、升版编号等格式

## 20. 当前系统状态判断

### 20.1 已经比较稳定的部分

- API 基础契约
- 交付主链
- 纠错主链
- 文档生成主链
- AutoCAD 主路径与资源受管
- Group + shared prep
- 终端部署包构建

### 20.2 仍在演进或未完全开放的部分

- `audit_replace mode=replace`
- 完整后端并发调度
- 更完善的系统环境管理页
- 某些复杂 DWG 的 layer 适配策略

### 20.3 当前维护原则

1. 先看规范，再改代码。
2. 先跑已有测试，再补新测试。
3. 图框问题先查 layer 与候选链，不要直接改 ROI。
4. 文档问题先查 `DocContext` 和派生值，不要直接怪模板。
5. AutoCAD 问题先看槽位资源与 task/result/log，再看 Bridge 实现。

## 21. 建议的接手动作

如果你是新的维护者，建议第一轮先做下面几件事：

1. 跑 `GET /api/system/health`，确认本机依赖状态。
2. 跑 `tools/run_test_stage.ps1 -Stage quick`。
3. 阅读一个真实 job 目录下的 `job.json` 和 `manifest.json`。
4. 用 `tools/run_frame_detector.py` 跑一个真实样本，理解 `frames` 与 `sheet_sets`。
5. 用前端本地 dev server 提交一次交付任务和一次纠错任务。

做到这一步，后续不管是改前端、改后端、改 CAD 还是改部署，都会更稳。
